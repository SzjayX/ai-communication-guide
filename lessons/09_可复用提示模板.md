# 课程 09：建立可复用的提示模板

## 学习目标
- 理解模板的价值：从"每次重写"到"一次写好，终身复用"
- 掌握模板的三个层次和适用场景
- 学会设计参数化的、可维护的提示模板
- 能搭建自己的模板库并持续迭代

---

## 核心概念

### 1. 为什么要建模板？

**一个扎心的事实**：你可能每天都在重复写 80% 相同的指令。

想一想你上周的工作：
- 让 AI 帮你写邮件 → 每次都在描述"语气要专业但不要太正式"
- 让 AI 帮你审查代码 → 每次都在说"请检查安全问题、性能问题、可读性"
- 让 AI 帮你写周报 → 每次都在解释"我们团队的周报格式是这样的"

**你其实不是在写新指令，你是在凭记忆重写旧指令。** 而且每次重写的质量还不一样。

**模板解决的就是这个问题**：

```
没有模板：
每次从头写 → 质量不稳定 → 遗漏关键要求 → 反复修改 → 浪费时间
     ↑________________________________________________________↓

有模板：
拿模板 → 填参数 → 首次即高质量 → 持续优化模板 → 越来越好用
```

**量化的收益**：

| 维度 | 无模板 | 有模板 | 提升 |
|------|--------|--------|------|
| 写指令时间 | 5-10 分钟 | 30 秒（填参数） | **10-20x** |
| 输出质量 | 忽高忽低 | 稳定高质量 | **显著** |
| 迭代轮数 | 2-3 轮 | 0-1 轮 | **2-3x** |
| 团队共享 | 口口相传 | 文档化可传播 | **质的飞跃** |
| 知识沉淀 | 存在脑子里 | 存在文档里 | **永久资产** |

---

### 2. 什么是好模板？

**好模板的公式**：

```
好模板 = 好指令 + 清晰的参数 + 使用说明 + 示例 + 迭代记录
```

拆解开来看：

| 要素 | 作用 | 缺了会怎样 |
|------|------|-----------|
| **好指令**（RTBSF 骨架） | 模板的核心，定义任务 | 模板没有灵魂 |
| **清晰的参数** | 告诉使用者哪些可以改 | 不知道怎么适配自己的场景 |
| **使用说明** | 什么场景适用，什么不适用 | 用错场景，效果差 |
| **示例**（正例+反例） | 展示正确用法和输出 | 使用者不知道"对"长什么样 |
| **迭代记录** | 记录了为什么这样设计 | 过时了没人敢改 |

### 3. 模板的三个层次

不同复杂度对应不同的模板设计方式。

---

#### 第 1 层：基础模板（复用率最高，占日常 60%）

**适用场景**：频繁重复的标准化任务

**特点**：
- 参数少（2-5 个）
- 结构固定
- 拿来即用，几乎不需要改

**案例：代码审查模板**

```markdown
# 【模板】代码审查

## 指令

你是 {role}，请审查以下代码。

审查维度：
1. 安全问题（注入、泄露、权限）
2. 性能问题（复杂度、冗余查询、内存）
3. 可读性和可维护性
4. 错误处理

## 参数

| 参数 | 说明 | 示例 |
|------|------|------|
| {role} | 审查者的角色 | "资深 Python 后端工程师，有 10 年安全审计经验" |
| {focus} | 重点关注领域 | "安全" / "性能" / "可读性" / "全面审查" |
| {context} | 代码的背景信息 | "这是支付模块的核心函数，每天处理 10 万笔交易" |

## 输出格式

对每个问题：严重度（🔴🟡🟢）→ 位置 → 风险/影响 → 修复建议

## 示例

✅ 好的审查：
---
🔴 高 | 第 45 行：SQL 字符串拼接 → 注入风险
修复：使用参数化查询 `cursor.execute(sql, params)`
---

❌ 不好的审查：
---
"代码有一些安全问题，需要改进"
（没有具体位置和修复方案，无法执行）
---

## 使用场景
✅ 提交代码前的自检
✅ 团队 Code Review 的辅助
❌ 不适合：架构级别的设计审查
```

**这个模板的价值**：任何人（包括你自己 3 个月后）拿到就能用，输出质量有保障。

---

#### 第 2 层：高级模板（复用率中等，占日常 30%）

**适用场景**：有多个可变维度的复杂任务

**特点**：
- 参数较多（5-10 个）
- 有多维度选择
- 需要使用者做一些决策

**案例：技术文档生成模板**

```markdown
# 【模板】技术文档生成

## 使用前选择

请在以下维度中做出选择：

### 维度 1：文档类型（选一个）
□ 快速入门 — 目标：新用户，深度：浅，结构：安装→用法→示例
□ API 参考 — 目标：开发者，深度：深，结构：端点→参数→响应→示例
□ 架构设计 — 目标：技术决策者，深度：深，结构：背景→方案→决策→风险
□ 运维手册 — 目标：运维团队，深度：中，结构：部署→配置→监控→排错

### 维度 2：受众技术水平（选一个）
□ 初级 — 需要解释基础概念，避免术语
□ 中级 — 可使用行业术语，略过基础
□ 高级 — 可深入技术细节，讨论权衡

### 维度 3：语言要求（选一个）
□ 中文 + 正式 — 企业内部文档
□ 中文 + 口语 — 开发者博客
□ 英文 + 正式 — 开源项目文档

## 指令（根据选择自动组合）

你是 {根据文档类型选择角色}，请为 {受众水平} 的技术人员写一份 {文档类型}。

## 各类型的具体要求

### 如果选了"快速入门"：
- 先展示一个"能跑起来"的最简示例
- 再解释每一步
- 最后给出"下一步"的指引
- 长度：500-800 字

### 如果选了"API 参考"：
- 列出所有端点
- 每个端点：HTTP 方法、路径、参数表、响应结构、一个 curl 示例
- 标注哪些参数是必填/可选
- 标注可能的错误码和含义

### 如果选了"架构设计"：
- 先描述业务背景和约束
- 给出推荐方案的架构图（用文字描述）
- 对比至少 2 个备选方案
- 分析每个方案的风险和应对策略

## 示例

[根据选择的维度组合，提供对应的正例和反例]

## 常见问题

Q：生成的文档太长怎么办？
A：在指令中加上"每个端点用一个 3 行内的示例即可"。

Q：需要更详细怎么办？
A：在指令中加入"请给出至少 5 个实际使用案例"。
```

**这个模板的价值**：把复杂的"因人而异"变成了结构化的选择题。即使团队新人也能生成符合标准的技术文档。

---

#### 第 3 层：框架级模板（复用率最低，但影响力最大）

**适用场景**：组织级别的关键工作流，如 AI Agent 设计

**特点**：
- 不只一个指令，是一个完整的工作流
- 包含设计哲学和原则
- 通常需要培训和讨论才能用好

> 这一层的详细内容将在课程 13（实战项目：构建 AI Agent）中深入展开。这里先建立概念。

### 4. 从零到一：手把手建一个模板

让我们用一个真实的例子，完整走一遍建模板的过程。

#### 场景：每周写项目周报

**Step 1：找一个你反复执行的任务**

你每周五下午都要写周报，每次都要跟 AI 解释：
- 我们团队是做什么的
- 周报的格式是什么
- 哪些该写、哪些不该写
- 语气应该怎样

这些信息每次都在你的指令里重复出现。

---

**Step 2：提取"不变"和"可变"的部分**

```
不变的部分（写进模板）：
├── 角色定义："你是我团队的项目助理"
├── 周报结构：本周进展 → 遇到的问题 → 下周计划 → 需要支持
├── 格式要求：Markdown，每条进展用一句话概括要点
├── 语气要求：简洁务实，不粉饰，不隐瞒
└── 长度要求：每个部分 3-5 条，总字数 300-500 字

可变的部分（设计为参数）：
├── {项目名}：这周在做什么项目
├── {本周要点}：这周的关键事件（3-5 条）
├── {问题}：遇到的阻碍（1-3 条）
├── {下周计划}：下周的重点（3-5 条）
├── {需求}：需要谁做什么支持
└── {受众}：这份周报给谁看（主管 / 全团队 / 客户）
```

---

**Step 3：用 RTBSF 写骨架指令**

```
【角色】
你是 {项目名} 的项目助理，负责帮团队撰写清晰、务实的周报。

【任务】
根据我提供的要点，生成一份结构化的项目周报。

【背景】
- 周报的受众是 {受众}
- 团队文化：诚实透明，有问题直接说，不藏不掖
- 周报用于：同步进展 + 暴露风险 + 协调资源

【步骤】
1. 以"本周关键词"开头（2-3 个词概括本周）
2. 按"进展→问题→计划→支持"组织内容
3. 每条要点用一句话概括，先写结论再写细节
4. 检查是否有遗漏的关键信息

【格式】
Markdown，结构如下：
## 本周关键词
## 📈 本周进展
## ⚠️ 遇到的问题
## 📋 下周计划
## 🤝 需要支持
```

---

**Step 4：加入示例**

```
✅ 好的周报风格：
---
## 本周关键词：性能优化、用户反馈、技术债清理

## 📈 本周进展
- 核心 API 响应时间从 2.3s 降到 0.8s（详情见技术方案文档）
- 整理了 127 条用户反馈，筛选出 15 个高优先级需求
- 完成登录模块的代码重构，单元测试覆盖率 78% → 92%

## ⚠️ 遇到的问题
- 数据库迁移脚本在预发布环境失败两次，预计延迟 1 天
- 后端同事下周休假，需要有人接手他的代码审查

## 📋 下周计划
- 完成用户反馈中 5 个高优先级需求的开发
- 灰度发布性能优化版本给 10% 用户
- 完成数据库迁移并验证

## 🤝 需要支持
- 需要运维协助预发布环境的数据库权限问题
---

❌ 不要这样写：
---
"本周我们做了很多工作，有很多进展。也遇到了一些小问题，
但很快就解决了。下周我们会继续努力，把剩下的工作做完。"
（缺乏具体信息，读者无法判断进度和风险，等于没写）
---
```

---

**Step 5：编写使用说明**

```
## 📖 使用说明

### 适用场景
✅ 项目进展汇报（给主管/团队）
✅ 跨部门同步（给其他团队负责人）

### 不适用场景
❌ 给客户的正式交付报告（需要更正式格式）
❌ 日报（太详细，换个模板）

### 使用步骤
1. 在项目笔记里收集本周的关键事件（5 分钟）
2. 把事件填入下面参数区的对应部分
3. 把完整模板发给 AI
4. 检查 AI 输出，通常 0-1 轮微调即可

### 优化技巧
- 如果 AI 的语气太正式：把受众改为"友好的团队成员"
- 如果 AI 遗漏了某个要点：在参数里显式标注"必须包含"
- 如果 AI 输出太短：增加每个部分的目标条目数
```

---

**Step 6：存起来，标注频率，记录迭代**

```
## 📊 模板信息

| 项目 | 内容 |
|------|------|
| 模板名称 | 项目周报生成 |
| 创建日期 | 2026-06-04 |
| 复用频率 | 🔴 高频（每周） |
| 分类 | 商务沟通 |
| 标签 | #周报 #项目管理 #汇报 |

## 🔄 迭代记录

| 版本 | 日期 | 改动 | 效果 |
|------|------|------|------|
| v1.0 | 2026-05-15 | 初版 | 需要 2 轮迭代 |
| v1.1 | 2026-05-22 | 加了反例 | 降低到 0-1 轮 |
| v1.2 | 2026-06-04 | 加了"本周关键词" | 读者反馈更容易把握重点 |
```

### 5. 参数设计的黄金法则

模板的核心难点是：**什么该参数化，什么该写死？**

| 法则 | 说明 |
|------|------|
| **每次都在变的 → 参数化** | 项目名、日期、具体数据、受众 |
| **很少变的 → 写死** | 团队文化、格式结构、质量标准 |
| **可选但影响大 → 可选参数** | 语气、长度、深度、语言 |
| **需要保护质量 → 给默认值** | 不给默认值的使用者可能乱填 |

**参数化的常见错误**：

```
❌ 过度参数化：
"以 {tone} 的语气写一份 {document_type}，长度为 {length}，
语言为 {language}，受众为 {audience}，包含 {include_sections}，
不包括 {exclude_sections}，重点是 {focus}，使用 {examples_count} 个例子…"
→ 填参数的时间比直接写指令还长

✅ 合理的参数化：
"以 {tone} 的语气写一份 {document_type}"
（只保留每次一定要改的 2 个参数，其余写死在模板里）
```

### 6. 搭建你的模板库

#### 选一个存储方式

| 方式 | 适合 | 优点 | 缺点 |
|------|------|------|------|
| **GitHub 仓库** | 个人+团队 | 版本控制、协作、公开分享 | 需要 Git 基础 |
| **Notion / 飞书** | 团队 | 易编辑、富文本、分享方便 | 没有版本控制 |
| **本地文件夹** | 个人 | 零成本、完全掌控 | 不易分享 |
| **Claude Projects** | Claude 用户 | 和 AI 深度集成 | 工具绑定 |

**推荐组合**：GitHub（存储 + 版本控制）+ 本地文件夹（快速访问）

#### 标记复用频率，决定优化顺序

```
🔴 高频（每周用多次） → 优先优化，值得花时间打磨
🟡 中频（每月用几次） → 定期维护，积累到一定反馈再改
🟢 低频（偶尔用） → 能用就行，不花太多精力
```

**原则**：先优化你每天用的那 3 个模板。它们的质量决定了你 80% 的 AI 产出。

### 7. 模板的维护与迭代

模板不是写完了就扔那。它会**老化**：

- 你的工作方式变了，模板没跟上
- AI 模型升级了，老指令不再是最优的
- 团队需求变了，旧模板不适用了

#### 维护节奏

| 频率 | 动作 |
|------|------|
| **每次使用后** | 记一笔：哪里需要调？调了什么？为什么？ |
| **每月** | 回顾高频模板的使用记录，决定是否改进 |
| **每季度** | 清理：删掉不用的，合并重复的，更新过时的 |
| **每年** | 大版本升级：根据一年积累的经验重构模板 |

#### 迭代的触发条件

如果出现以下情况，就该迭代了：

```
□ 同一个参数每次都要改成某个值 → 把那个值设为默认
□ 某个参数从来没人改过 → 考虑写死它
□ AI 的输出经常在某个地方出问题 → 加强那部分的约束
□ 出现了新的使用场景 → 评估是否需要新建模板或扩展现有模板
□ 模型升级后输出风格变了 → 重新校准参数和示例
```

## 关键要点总结

1. **模板是个人经验 → 可复用资产的转化器**：不建模板，经验永远只存在你脑子里
2. **三个层次**：基础模板（60%日常）→ 高级模板（30%日常）→ 框架模板（10%，影响最大）
3. **好模板五要素**：好指令 + 清晰参数 + 使用说明 + 示例 + 迭代记录
4. **参数设计核心**：每次变的参数化，很少变的写死，不给默认值的给出默认值
5. **先建高频的**：优先打磨每天用的 3-5 个模板，它们决定你 80% 的产出
6. **模板需要维护**：定期回顾、更新示例、清理过时内容
7. **分享即学习**：把模板分享给团队，收集反馈，这是最好的改进方式

## 实践活动

### 练习 1：发现已有的隐式模板（15 分钟）
回顾你最近一周的 AI 使用记录，找出 3 个反复执行的任务。对每个任务回答：
- 这个任务我做了几次？
- 每次写指令的哪些部分是一样的？
- 哪些部分每次都在改？

→ 这就是你的"隐式模板"，把它们显式化。

### 练习 2：建第一个模板（30 分钟）
选一个练习 1 中找到的任务，按以下步骤建模板：
1. 提取不变的部分（写死）和可变的部分（参数化）
2. 用 RTBSF 写骨架指令
3. 加一个正例和一个反例
4. 写使用说明（适用/不适用场景 + 优化技巧）
5. 保存到你的模板库里

### 练习 3：测试你的模板（20 分钟）
用你刚建的模板，完成 3 次不同参数组合的任务。记录：
- 每次需要多少额外调整？
- 输出质量是否稳定？
- 还有哪些地方可以改进？

### 练习 4：搭建模板库骨架（20 分钟）
1. 创建一个文件夹作为模板库的根目录
2. 建立 3-5 个分类文件夹（参考本课的推荐结构）
3. 把练习 2 的模板放进去
4. 列一个清单：你接下来想建哪些模板？
