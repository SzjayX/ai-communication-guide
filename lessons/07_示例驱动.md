# 课程 07：示例驱动（正例和反例）

## 学习目标
- 理解为什么示例比文字描述更有效
- 掌握正例、反例、对比示例三种驱动方式
- 学会编写高质量的示例来指导 AI 输出
- 能够在实际指令中灵活运用示例

---

## 核心概念

### 1. 为什么示例这么重要？

**核心理念：语言是模糊的，但例子是清晰的。**

一个残酷的事实：你再怎么用文字描述，AI 也很难完全理解你脑海中那个"完美输出"的样子。但一个具体的例子，可以瞬间传递 100 行文字无法表达的信息。

#### 一个对比实验

假设你想让 AI 写产品评测。看看两种方式的差异：

**方式 A：纯文字描述**
```
请写一篇产品评测，要专业、客观、有深度，
包含优缺点分析，最后给出购买建议。
字数控制在 500 字左右。
```

**方式 B：文字描述 + 示例**
```
请写一篇产品评测，参考以下示例的风格和结构：

✅ 参考示例：
---
产品：AirPods Pro 2
总体评分：8.5/10

优点：
• 降噪效果突出——通勤途中地铁噪音降低约 80%
• 佩戴舒适——连续佩戴 3 小时无不适感
• 与苹果生态无缝集成——iPhone/iPad/Mac 自动切换

缺点：
• 售价 1799 元，门槛较高
• 不支持安卓设备的空间音频等高级功能
• 单次续航仅 6 小时，重度使用需中途充电

购买建议：如果你是 iPhone 用户且经常通勤/出差，强烈推荐。
如果你使用安卓手机或预算有限，建议考虑其他品牌。
---
```

AI 从示例中学到什么？
- 结构：评分 → 优点 → 缺点 → 购买建议
- 深度：每个优缺点都有具体解释和数据
- 风格：客观冷静，不夸大不贬低
- 细节：优缺点数量相当（3 个 vs 3 个），显得公正

**这就是示例的威力。** AI 不是从你的"描述"中学习，而是从你的"例子"中**学习模式**。

---

### 2. 示例为什么有效的科学解释

#### 大语言模型的本质：模式匹配

大语言模型的核心能力是**模式识别与模式复现**。当你给它一个示例时：

1. **模式提取**：AI 从示例中提取结构模式、语言风格、思维方式
2. **模式泛化**：AI 将这个模式应用到你的新任务上
3. **模式约束**：示例划定了一个"范围"，防止 AI 跑偏

**类比**：教一个人做菜
- ❌ "做一道好吃的川菜"（太模糊）
- ✅ "来看我做一次回锅肉，注意火候和调料顺序"（具体可模仿）

这就是为什么 Few-shot Prompting（少样本提示）在学术界被证明是最有效的提示技术之一。

### 3. 三种示例驱动方式

#### 方式一：正例（Positive Examples）— "我要的是这样"

正例告诉 AI：**这就是我期望的样子**。

**正例的核心要素**：

| 要素 | 说明 | 示例 |
|------|------|------|
| 结构 | 输出的组织方式 | 先评分、再列优缺点、最后建议 |
| 深度 | 信息的详细程度 | 每个优点附带数据支撑 |
| 风格 | 语言和语调 | 客观冷静 vs 热情推销 |
| 长度 | 输出的篇幅 | 500 字 vs 50 字摘要 |

**正例的编写原则**：

1. **和最终需求同等复杂度**
   - 如果你的任务产出是 2000 字分析报告，给一个 200 字的正例是没用的
   - 正例应该尽可能接近你期望的实际输出

2. **覆盖关键变化情况**
   - 如果任务有多种可能的输入，提供 2-3 个不同类型的正例
   - 例如：评测手机的正例 + 评测软件的正例 + 评测服务的正例

3. **标注清楚"这是参考"**
   - 用 ✅、📝、"参考示例"等标记明确标注
   - 避免 AI 把示例当作指令的一部分

**正例示例（代码审查场景）**：

```
✅ 我期望的审查报告格式：
---
【文件】src/services/userAuth.js
【总体评分】6.5/10

【安全问题】🔴 高优先级
1. 第 23 行：密码明文存储在 localStorage
   → 风险：任何有浏览器访问权限的人都能获取
   → 修复：使用 httpOnly cookie + session token

2. 第 45 行：SQL 查询使用字符串拼接
   → 风险：SQL 注入攻击
   → 修复：使用参数化查询

【性能问题】🟡 中优先级
1. 第 78 行：在循环内执行数据库查询（N+1 问题）
   → 影响：100 个用户 = 101 次查询
   → 修复：使用 JOIN 或 IN 批量查询

【代码质量】🟢 建议
1. 函数 authenticateUser 长达 120 行，建议拆分
2. 缺少错误处理，网络异常时会崩溃
---
```

#### 方式二：反例（Negative Examples）— "不要这样做"

反例告诉 AI：**这是禁区，不要越界**。

**反例的作用**：
- 划定边界：明确什么是不接受的
- 防止跑偏：AI 有时会自作主张地"改进"
- 强调重点：通过对比让你真正在意的点更突出

**反例的编写原则**：

1. **反例必须和正例属于同一类别**
   - ❌ 用"学术论文"作为"营销文案"的反例 → 毫无意义
   - ✅ 用"糟糕的营销文案"作为"好的营销文案"的反例 → 对比清晰

2. **反例应该体现最常见的错误**
   - 想一想 AI 通常会犯什么错？
   - 把这些错误明确展示出来

3. **反例要有针对性，不要泛泛而谈**
   - ❌ "不要写得不好"（废话）
   - ✅ "不要只列功能不写好处"（具体可操作）

**反例示例（继续代码审查场景）**：

```
❌ 不好的审查报告：
---
代码有一些安全问题，需要改进。
总体来说代码质量一般，建议优化。
---
↑ 这个反例的问题：
• 没有具体指出哪里有问题
• 没有给出优先级
• 没有提供修复方案
• 笼统的评语没有任何实用价值
```

---

#### 方式三：对比示例（Side-by-Side）— "对的是这样，错的是那样"

将正例和反例放在一起对比，是最高效的示例驱动方式。

**对比示例的结构**：

```
✅ 好的做法：
[正例内容]

❌ 不好的做法：
[反例内容]

📝 关键差异：
• 差异 1：好的有 X，不好的缺 X
• 差异 2：好的用 Y 方式，不好的用 Z 方式
• 差异 3：好的深度到 A 级别，不好的只到 B 级别
```

**为什么要加"关键差异"说明？**

示例虽然直观，但有时候 AI 可能学到了你不想让它学的东西。明确列出关键差异，确保 AI 关注的维度是正确的。

**对比示例（营销文案场景）**：

```
✅ 好的文案：
---
厌倦了手动整理数据？Claude 帮你从 500 行 Excel 直接得到洞察。
🎯 5 秒看懂趋势 | 📊 零代码操作 | 🚀 比手工快 100 倍
→ 立即试用，每月免费额度 100 万 tokens
---

❌ 不好的文案：
---
Claude 是 Anthropic 开发的先进大语言模型，采用最新的
Transformer 架构和强化学习技术，拥有强大的自然语言处理
能力，可以帮助企业提升效率、降低成本、驱动增长。
---

📝 关键差异：
• 好的：直接对用户说话（"你"）| 不好的：在介绍产品（"Claude 是"）
• 好的：讲好处（"5 秒看懂趋势"）| 不好的：讲功能（"Transformer 架构"）
• 好的：有行动号召（"立即试用"）| 不好的：用抽象词汇堆砌
• 好的：短句快节奏 | 不好的：长句读起来累
• 好的：emoji 增加亲和力 | 不好的：冷冰冰没有人情味
```

### 4. 示例驱动的四大黄金法则

| 法则 | 含义 | 反面教材 |
|------|------|---------|
| **多样性** | 提供 2-3 个正例，覆盖不同的输入类型和场景 | 只给一个例子，AI 无法泛化 |
| **细节完整** | 示例应该是完整的、真实的，不是骨架或模板 | 给"评测：产品名 / 评分：X / 优点：..." 这种空壳 |
| **同类对比** | 正反例必须是同一类任务、同一领域 | 用写诗的正例来教 AI 写代码 |
| **标注清晰** | 用 ✅❌ 或明确的标签区分正反例 | 混在一起让 AI 自己猜 |

#### 法则详解

**法则 1：多样性**

```
✅ 提供 3 个不同类型的正例：
- 正例 1：评测硬件产品（手机）
- 正例 2：评测软件产品（App）
- 正例 3：评测服务产品（在线课程）
→ AI 学会的是"评测"这个模式，而不是"评测手机"这个特例

❌ 只给 1 个正例：
- 正例：评测 iPhone
→ AI 可能只会评测手机，换到评测软件就变形
```

**法则 2：细节完整**

```
✅ 完整的正例：
展示一段 300 字的实际产品评测，包含评分、优缺点、建议

❌ 骨架式正例：
"产品：[名称] / 评分：[数字] / 优点：[列表] / 缺点：[列表]"
→ AI 学会的是"填空"，而不是"写作"
```

**法则 3：同类对比**

```
✅ 合理的对比：
好的产品评测 vs 差的产品评测（同领域、同任务）

❌ 不合理的对比：
好的产品评测 vs 差的诗歌（完全不同领域，无法对比）
```

**法则 4：标注清晰**

```
✅ 清晰标注：
✅ 正例：...
❌ 反例：...
📝 差异：...

❌ 含糊标注：
例子 1：...
例子 2：...
（AI 分不清哪个是你想要的）
```

### 5. 完整示例：一个指令的进化

让我们看一个完整的进化过程。

**版本 1：无示例（效果差）**
```
请写一个 Python 函数的文档注释。
```

AI 输出：
```python
def calculate(x, y):
    """Calculate the result."""
    return x + y
```
↑ 基本没用。

**版本 2：加入正例（效果好多了）**
```
请写一个 Python 函数的文档注释。

✅ 参考示例：
---
def fetch_user_orders(user_id: str, start_date: str, end_date: str) -> list[dict]:
    """获取指定用户在给定时间范围内的订单列表。

    Args:
        user_id: 用户唯一标识符，格式为 "USR-XXXXX"
        start_date: 查询起始日期，格式 "YYYY-MM-DD"（含当日）
        end_date: 查询截止日期，格式 "YYYY-MM-DD"（含当日）

    Returns:
        订单字典列表，每个订单包含:
        - order_id (str): 订单号
        - amount (float): 金额（元）
        - status (str): 状态 ("pending" | "completed" | "cancelled")
        - created_at (str): 创建时间 (ISO 8601)

    Raises:
        ValueError: user_id 格式不正确时
        ConnectionError: 数据库连接失败时

    Examples:
        >>> fetch_user_orders("USR-00123", "2026-01-01", "2026-01-31")
        [{"order_id": "ORD-456", "amount": 299.0, "status": "completed", ...}]
    ---
```

现在 AI 知道了：类型提示、参数说明格式、返回值详细描述、异常说明、使用示例。

**版本 3：加入反例（效果最佳）**
```
写 Python 函数的文档注释。

✅ 参考示例：
[上面的完整正例]

❌ 不要这样：
---
def calculate(x, y):
    """Calculate the result.
    
    Args:
        x: the first parameter
        y: the second parameter
    
    Returns:
        the result
    """
    return x + y
---

📝 关键要求：
• 每个参数必须说明类型、格式、含义，不只是名称
• 返回值必须列出所有字段及其类型和含义
• 必须包含 Raises 和 Examples 部分
```

### 6. 不同场景的示例策略

#### 场景 1：创意类任务（文案、故事、设计）

**策略**：多正例，展示风格多样性

```
任务：写品牌故事

✅ 风格 A — 情感共鸣型：
[示例：从创始人故事切入，打动人心]

✅ 风格 B — 数据说服型：
[示例：以行业数据和洞察开篇，建立信任]

✅ 风格 C — 用户故事型：
[示例：以典型用户的使用场景为主线]
```

#### 场景 2：技术类任务（代码、方案、审查）

**策略**：正例 + 反例，强调规范

```
任务：写技术方案

✅ 好的方案应有的结构：
[完整结构示例]

❌ 不好的方案（常见问题）：
[缺少关键部分的反例]

📝 必须包含：需求分析、方案对比、架构设计、风险评估、实施计划
```

#### 场景 3：分析类任务（调研、评估、对比）

**策略**：对比例子，展示深度要求

```
任务：竞品分析

✅ 有深度的分析：
[层层递进的分析结构、数据支撑、洞察结论]

❌ 肤浅的分析：
[只是列表对比、没有洞察、没有建议]

📝 关键差异：不止于"是什么"，更要回答"所以呢"和"然后呢"
```

### 7. 示例数量的选择

| 任务复杂度 | 推荐示例数 | 理由 |
|-----------|-----------|------|
| 简单任务 | 1 个正例 | 任务本身简单，一个例子足够 |
| 中等任务 | 2-3 个正例 + 1 个反例 | 需要展示不同情况，划定边界 |
| 复杂任务 | 3-5 个正例 + 2-3 个反例 | 输出多变，需要大量约束 |
| 持续合作 | 建立示例库 | 长期使用，标准化输出 |

**注意**：示例不是越多越好。过多的示例会：
- 占用上下文窗口，挤掉其他重要信息
- 让 AI 过度模仿，失去灵活性
- 增加你的编写负担

**黄金原则**：用最少的示例，传递最多的信息。

## 关键要点总结

1. **语言是模糊的，例子是清晰的**：一个具体示例胜过一百行文字描述
2. **三种方式组合使用**：正例展示目标 + 反例划定边界 + 对比强调差异
3. **四大黄金法则**：多样性、细节完整、同类对比、标注清晰
4. **示例的质量决定 AI 输出的质量**：骨架式的示例 = 骨架式的输出
5. **反例和正例同等重要**：只给正例，AI 不知道边界在哪里
6. **示例是对 RTBSF 框架的补充**：RTBSF 定义"要做什么"，示例展示"做成什么样"

## 实践活动

### 练习 1：示例诊断（10 分钟）
找一个你之前写的指令，检查：
- [ ] 有没有提供示例？
- [ ] 如果有，是正例还是反例？
- [ ] 示例是否完整且有细节？
- [ ] 缺少什么类型的示例？

### 练习 2：为旧指令添加示例（20 分钟）
选一个你常用的指令，给它添加：
1. 1-2 个高质量的正例
2. 1 个反例（展示常见错误）
3. 关键差异说明

然后对比添加示例前后的 AI 输出质量。

### 练习 3：设计"示例驱动"指令（30 分钟）
用示例驱动的思路，从头设计一个新的指令，要求：
1. 先用 RTBSF 框架搭建骨架
2. 再添加 2-3 个正例（覆盖不同场景）
3. 添加 1-2 个反例（预防常见问题）
4. 添加关键差异说明
5. 测试 AI 输出并与无示例的版本对比

### 练习 4：建立示例片段库（持续）
开始收集你在日常工作中遇到的：
- 优秀输出 → 作为未来指令的正例
- 糟糕输出 → 作为未来指令的反例
- 典型场景 → 作为示例多样性的素材
