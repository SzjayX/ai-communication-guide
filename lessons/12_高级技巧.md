# 课程 12：高级技巧

## 学习目标
- 掌握思维链提示（Chain-of-Thought），让 AI 展示推理过程而非直接给结论
- 精通角色扮演技巧，通过构建完整的人物设定来精准控制 AI 的视角和风格
- 学会约束驱动设计，用精确的边界条件替代模糊的愿望
- 理解多轮对话的系统化推进策略，不再"改一步看一步"
- 掌握让 AI 自我反思和自检的技巧，让 AI 帮你找自己输出的问题
- 能根据任务特点灵活选择和组合五大技巧

---

## 核心概念

### 1. 高级技巧的定位

先回顾一下你到目前为止学到了什么：

```
第一阶段（认知）   → 你知道 AI 能做什么、不能做什么
第二阶段（基础）   → 你能用 RTBSF 写出结构清晰的指令
第三阶段（进阶）   → 你会用示例、迭代、模板，还会系统化思考、避开常见错误
```

**那"高级技巧"高在哪里？**

基础技巧让你"把话说清楚"，高级技巧让你"控制 AI 怎么想"。

```
基础：告诉 AI 你要什么 —— "帮我写一篇技术博客"
高级：控制 AI 的思考过程 —— "先用思维链梳理逻辑，
        再以资深博主的身份写，每段不超过 150 字，
        完成后自查是否符合技术博客的 5 个标准"
```

**这不是量的区别，是质的区别。** 基础技巧解决"AI 听懂了吗"，高级技巧解决"AI 想到位了吗"。

## 技巧 1️⃣：思维链（Chain-of-Thought）

### 什么是思维链？

思维链的核心思想非常简单：**不要让 AI 直接跳到结论，要求它把思考过程写出来。**

这背后有一个认知科学原理：当人类被要求"一步步思考"时，我们的推理质量显著提高。AI 也一样——写出推理过程能让它避免跳步、减少遗漏、降低幻觉。

**一个直观的对比**：

```
❌ 直接问：
"一个长方形长 12cm、宽 8cm。如果长增加 3cm、宽减少 2cm，面积变化多少？"

AI 可能直接回答："增加了 X 平方厘米"——数字可能对，也可能错，但你看不到过程。

✅ 思维链问法：
"请一步步推理：
1. 原始面积是多少？
2. 新的长和宽各是多少？
3. 新面积是多少？
4. 面积差是多少？
5. 这意味着面积增加还是减少？"

AI 会输出每一步的计算，你可以检查每一步是否正确。
```

### 思维链的三种用法

#### 用法 1：显式步骤（最简单）

直接列出推理步骤，让 AI 按顺序执行：

```
"请按以下步骤分析这段代码的性能问题：

Step 1：识别这段代码的功能和目的（一句话概括）
Step 2：逐行标注每行的时间复杂度
Step 3：找出最慢的 3 个操作
Step 4：对每个慢操作给出优化建议
Step 5：估算优化后的整体性能提升幅度

代码：
[粘贴代码]"
```

**适用场景**：问题结构清晰，你知道正确的推理路径。

---

#### 用法 2：示例引导（Few-Shot CoT）

给 AI 一个类似问题的推理过程作为示例，让它模仿：

```
"我现在要你分析一些技术方案的可行性。请参考以下示例的分析方式：

【示例】
问题：要不要用微服务架构重构现有的单体应用？
推理过程：
1. 当前状态分析：单体应用，3 个模块，2 万行代码，日均请求 5 万次
2. 微服务的好处：独立部署、技术栈灵活、团队可并行开发
3. 微服务的代价：运维复杂度×3、网络延迟增加、数据一致性更难保证
4. 当前是否需要：日均 5 万请求量，单体完全够用，团队只有 5 个人
5. 结论：目前不需要。当日均请求超过 50 万或团队超过 15 人时再考虑。

请按同样的推理结构，分析以下问题：
[你的问题]"
```

**适用场景**：你有一类反复要做的问题，想统一推理风格和深度。

---

#### 用法 3：零样本触发（Zero-Shot CoT）

只加一句"让我们一步步思考"就能显著提升推理质量。这是最轻量的用法：

```
"分析这个产品为什么用户留存率低。让我们一步步思考。"
```

**研究发现**：仅仅加上"Let's think step by step"（让我们一步步思考），就能让 AI 的推理准确率提升 10%-40%，尤其在数学和逻辑推理任务上。

**适用场景**：问题不算特别复杂，但想让 AI 多花一点"思考时间"。

---

### 思维链的适用判断

| 场景 | 适合度 | 原因 |
|------|--------|------|
| 数学/逻辑推理 | ⭐⭐⭐⭐⭐ | 推理链清晰，验证容易 |
| 代码分析与调试 | ⭐⭐⭐⭐⭐ | 执行路径分析天然需要步骤 |
| 多方案比较 | ⭐⭐⭐⭐ | 每个方案独立的推理链 |
| 策略/规划 | ⭐⭐⭐⭐ | 推理→结论的结构清晰 |
| 信息查询 | ⭐ | 不需要推理，直接检索即可 |
| 创意生成 | ⭐⭐ | 创意不需要逻辑推理，但可以用"灵感展开" |
| 简单翻译 | ⭐ | 直接翻译即可，不需要推理 |

---

### 思维链的常见错误

```
❌ 步骤太粗糙：
"第一步：分析；第二步：总结"
→ 没有实际的推理指引

✅ 步骤具体可执行：
"第一步：算初始值；第二步：算变化量；第三步：算最终值；第四步：对比"
→ 每一步该做什么很清楚

❌ 步骤太多（超过 7 步）：
→ AI 可能在中间丢失上下文

✅ 步骤控制在 5-7 步：
→ 关键推理节点，不是啰嗦的流水账
```

## 技巧 2️⃣：角色扮演（Role-Playing）

### 不只是"你是一个专家"

你可能已经在用角色扮演了——"你是一个资深的 Python 工程师"。但这只是角色扮演最表面的用法。

**真正有效的角色扮演，是构建一个完整的人物设定**，包括：
- 身份背景（不只是一个职称）
- 知识结构（知道什么、不知道什么）
- 思维偏好（怎么思考问题）
- 语言风格（怎么说、用什么词）
- 价值观和偏见（看重什么、讨厌什么）

---

### 角色设定的五个维度

```
┌──────────────────────────────────────────────────────────┐
│              角色设定五维度                                │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  👤 身份与背景                                            │
│     ├─ 职业身份：不是"程序员"，是"有 8 年经验的           │
│     │   后端架构师，专注高并发系统"                        │
│     ├─ 经历背景：参与过什么项目、遇到过什么坑               │
│     └─ 知识边界：精通什么、了解什么、不懂什么               │
│                                                          │
│  🧠 思维模式                                              │
│     ├─ 思考习惯：先看全局再深入细节？还是从具体问题入手？    │
│     ├─ 决策偏好：偏好简单方案还是完备方案？                 │
│     └─ 风险态度：保守派还是激进派？                        │
│                                                          │
│  💬 语言风格                                              │
│     ├─ 词汇选择：用术语还是大白话？                        │
│     ├─ 句式偏好：长句还是短句？反问还是陈述？               │
│     ├─ 常用口头禅："实际上……""这里有个坑……"               │
│     └─ 禁用表达：不说什么话？                              │
│                                                          │
│  👁️ 关注焦点                                              │
│     ├─ 最看重什么？（安全性？性能？用户体验？成本？）        │
│     ├─ 最先注意到什么？（代码质量？产品逻辑？交互细节？）    │
│     └─ 最容易忽略什么？                                   │
│                                                          │
│  🎯 对受众的假设                                          │
│     ├─ 受众的知识水平：需要解释基础概念吗？                 │
│     ├─ 受众的关心点：他们最想知道什么？                     │
│     └─ 和受众的关系：导师？同事？服务者？                   │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 对比：浅层 vs 深层角色扮演

#### 浅层角色（作用有限）

```
"你是一个产品经理，帮我分析这个需求"
```

AI 会套用"产品经理"的通用模板，但分析结果可能和"你是一个工程师"差不太多。

#### 深层角色（效果显著）

```
"请你扮演以下角色来分析这个需求：

👤 身份：
你是一个在 SaaS 公司工作了 6 年的高级产品经理。
你经历过 3 个从 0 到 1 的产品，其中一个被收购，一个失败了。
你对 B2B 和 B2C 产品都有经验，但更擅长 B2B。

🧠 思维模式：
- 你习惯先问'这个需求解决了谁的什么问题'，再讨论方案
- 你偏好用数据做决策，对没有数据支撑的观点保持怀疑
- 你对功能膨胀有本能的警惕——见过太多产品死于'什么都想做'

💬 语言风格：
- 简洁直接，不喜欢包装和美化
- 经常反问：'如果做这个，我们不做哪个？'（因为资源有限）
- 用具体的数字和场景说事，避免'一般来说''通常情况'这种模糊表达

👁️ 关注焦点：
- 最关心：用户留存率 > 用户增长（先留住了再拉新）
- 先看：这个需求和公司战略是否对齐（不是所有好需求都该做）
- 容易忽略：技术实现的细节（需要工程师补充）

现在，请用这个角色的视角，分析这个新功能需求：
[需求描述]"
```

**这种深层角色设定的效果**：
- AI 的分析会有明确的视角和立场，而不是"各打五十大板"的中立分析
- 输出会有鲜明的个人风格，读起来像一个真实的人在跟你讨论
- 它能主动提出"你没想到的问题"——因为这个角色有自己的思维模式

---

### 角色扮演的三种实战模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **专家模式** | 扮演某领域专家，提供专业视角 | 技术分析、法律建议、医疗科普 |
| **受众模拟** | 扮演你的目标用户/读者，帮你检验内容 | 文案测试、教程审阅、产品验证 |
| **对立角色** | 扮演反对者/挑战者，找出你的盲点 | 方案评审、风险评估、辩论 |

**受众模拟示例**（特别有用但被低估）：

```
"现在你不是 AI，你是我们产品的目标用户。

用户画像：
- 姓名：小张，28 岁，后端开发工程师
- 工作：在一家 50 人的互联网公司，负责支付系统
- 痛点：每天要调试 20+ 个 API 接口，手动写测试用例很烦
- 技术水平：对 API 调试很熟，但对 AI 工具半信半疑
- 行为习惯：遇到新工具先看 GitHub Star 数，不到 1000 不看

请用小张的视角阅读以下产品介绍文案，然后告诉我：
1. 他读完后最想吐槽的 3 个点
2. 哪句话让他决定'试试看'？
3. 哪句话让他觉得'又在吹牛'？
4. 他会不会转发给同事？为什么？"

产品文案：
[粘贴你的文案]
```

## 技巧 3️⃣：约束驱动（Constraint-Based）

### 约束 vs 建议

自然语言有一个根本问题：**太灵活了。**

你说"写一篇有趣的文章"，AI 理解的"有趣"可能和你想要的"有趣"差了十万八千里。但如果换成约束：

```
"每 300 字必须出现一个让读者'等等，这有点意思'的反直觉事实。
不允许连续 3 段没有具体数据。结尾必须以一个新的待解决问题收尾，
让读者想点下一篇。"
```

效果完全不同。**约束不给 AI 模糊的发挥空间，它告诉 AI "边界在哪里"。**

---

### 约束的六种类型

```
┌──────────────────────────────────────────────────────────┐
│              六大约束类型                                  │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  📏 范围约束：规定大小、长度、数量                          │
│     "全文 800-1000 字，每段 100-150 字，不少于 5 个代码示例" │
│                                                          │
│  🚫 禁止约束：明确规定不能做什么                            │
│     "禁止使用'非常''很''极其'等程度副词；禁止用被动语态"     │
│                                                          │
│  ✅ 必需约束：规定必须包含什么                              │
│     "必须包含：1 个真实案例、1 组对比数据、1 个操作清单"     │
│                                                          │
│  🎨 风格约束：控制语言和表达方式                            │
│     "句式：主动语态为主。节奏：长句解释，短句强调。          │
│      禁用词：赋能、抓手、闭环、底层逻辑"                    │
│                                                          │
│  📐 结构约束：规定组织方式                                 │
│     "结构：问题(10%) → 分析(40%) → 方案(30%) → 风险(20%)"  │
│                                                          │
│  🔢 质量约束：规定验收标准                                 │
│     "每个观点必须有出处。代码示例必须可运行。               │
│      专业术语首次出现必须解释。"                            │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

---

### 约束驱动实战：代码审查

这是一个"约束密度"很高的指令示例：

```
"审查以下代码。约束：

📏 范围：
- 只关注安全问题和性能问题（不关注命名、格式、注释）
- 每个问题描述不超过 80 字

🚫 禁止：
- 禁止说'看起来不错'这类没有信息量的评价
- 禁止模糊描述（如'有安全问题'——要说具体什么问题）

✅ 必需：
- 每个问题必须标注：🔴高/🟡中/🟢低 严重度
- 每个问题必须给出具体的修复代码（不是建议，是可直接替换的代码）
- 如果这段代码没有问题，回答一个字：'过'

🎨 风格：
- 用祈使句（'将 X 改为 Y'），不用虚拟语气（'建议将 X 改为 Y'）
- 不解释为什么这是个问题，只给修复方案（受众是有经验的开发者）

代码：
[粘贴代码]"
```

**为什么约束驱动比"建议"更有效**：
- 建议："请写得专业一点" → AI 不知道"专业"的边界在哪
- 约束："禁用口语词、每段 3-5 句、必须有数据引用" → AI 知道精确的边界

---

### 约束密度的平衡

```
约束太少 → AI 自由发挥空间太大 → 输出不可控
约束恰当 → AI 在边界内发挥 → 输出质量稳定
约束太多 → AI 被绑住手脚 → 输出僵硬、不自然
```

**经验法则**：一个指令的约束数量控制在 5-8 条。少于 3 条可能太宽松，超过 10 条 AI 可能遗忘或顾此失彼。

如果你的约束确实很多（15+ 条），分优先级：

```
"约束（按重要性排列，如果冲突，优先满足排名靠前的）：
1. [最重要] 代码可直接运行
2. [重要] 响应时间 < 200ms
3. [重要] 错误处理覆盖所有异常
4. [次要] 代码注释覆盖关键逻辑
5. [次要] 变量命名遵循团队规范"
```

## 技巧 4️⃣：多轮对话优化（Conversation Engineering）

### 多轮对话的误区

大多数人这样做多轮对话：

```
第 1 轮：给个大致方向 → AI 输出
第 2 轮：嗯……不太对，你改改 → AI 输出
第 3 轮：还是不对……改一下第三段 → AI 输出
第 4 轮：好一点了，但……→ AI 输出
```

**问题**：每轮都是临时的、即兴的、反应式的。没有方向，没有目标，靠感觉推进。

**系统化的多轮对话**，是在开始之前就想好：我大概需要几轮？每轮的目标是什么？如何验证每轮是否达到了目标？

---

### 多轮对话的三种推进策略

#### 策略 1：逐层深入（Depth-First）

每轮在一个维度上加深：

```
第 1 轮：确定大方向
"先列出这个问题的 3 种可能解决方案，不展开，每种用 2 句话概述"

第 2 轮：选最有希望的方向深挖
"方案 B 看起来最有潜力。请详细展开方案 B：
- 实施步骤（分 3 个阶段）
- 每个阶段的时间和资源需求
- 可能遇到的最大风险"

第 3 轮：细节打磨
"方案 B 的 Phase 2 中，数据库迁移的部分风险最高。
请专门设计数据库迁移的详细方案，包括回滚策略"
```

**适用场景**：问题复杂，一层套一层，需要逐层剥开。

---

#### 策略 2：广度展开（Breadth-First）

先铺开再收窄：

```
第 1 轮：全面铺开
"从技术、产品、市场、团队、财务 5 个维度分别分析
这个项目是否可行。每个维度不超过 200 字"

第 2 轮：聚焦关键
"技术维度和市场维度都有重大风险。请深入分析这两个维度：
- 技术风险的最大 3 个来源和应对方案
- 市场风险的最大 3 个来源和应对方案"

第 3 轮：综合决策
"综合考虑以上分析，给出：Go / No-Go 建议，附带 3 个条件"
```

**适用场景**：需要全面评估，防止遗漏重要维度。

---

#### 策略 3：反复打磨（Iterative Refinement）

针对一个输出反复改进：

```
第 1 轮：出初稿
"按照我们之前讨论的 RTBSF 框架，生成初稿"

第 2 轮：针对性改进
"初稿的第三部分数据支撑不够。请补充：
- 至少 2 个具体的数据引用
- 1 个对比表格"

第 3 轮：全局优化
"整体检查：逻辑是否连贯？有没有前后矛盾？
优化过渡段，让文章读起来更流畅"

第 4 轮：终审
"这是最终版本。请自查：5 个核心卖点是否全部覆盖？
有没有超过 50 字的句子需要拆分？"
```

**适用场景**：对输出质量要求高，追求从 80 分到 95 分。

---

### 多轮对话的最佳实践

```
✅ 每轮有一个明确的目标
   "本轮目标是：确认方案的技术可行性"
   （而不是"再改改"）

✅ 每轮解决一个维度
   第 2 轮只改风格，第 3 轮只改结构，第 4 轮只加数据
   （而不是每次什么都调一点）

✅ 告诉 AI 保留什么
   "第二段和第四段很好，不要改。只调整第三段的逻辑顺序"
   （保护已经满意的部分）

✅ 多轮不等于无限
   设定轮数上限（3-5 轮），到了就取最好的一版

✅ 在重要任务中，第一轮先对齐标准
   "在开始写作之前，先确认你理解的要求是否和我一致"
   （避免写完了发现理解错了）
```

## 技巧 5️⃣：AI 自我反思（Self-Reflection）

### 让 AI 做自己的审查者

这是最被低估的高级技巧。核心思想：**AI 在输出之后，再让它以"审查者"的身份重新审视自己的输出。**

为什么有效？
- AI 的"生成模式"和"审查模式"是两种不同的状态
- 生成时追求流畅和完整，审查时追求准确和严谨
- 切换模式能发现第一遍没注意到的问题

---

### 自我反思的三种模式

#### 模式 1：事后自检（Post-hoc Review）

AI 生成内容后，追加一条要求它自检：

```
"现在，请切换身份，以一个挑剔的编辑的视角审查你刚才的输出。

检查清单：
□ 有没有事实性错误？（日期、数字、名称是否正确）
□ 有没有逻辑上的跳跃？（从 A 直接到 C，跳过了 B）
□ 有没有模糊的话？（'一般来说''某种程度上'这类表达）
□ 有没有不一致？（前面说 A，后面暗示非 A）
□ 读者最可能在哪里产生困惑？

找出问题后，直接告诉我需要修正的地方，
然后生成修正后的版本。"
```

---

#### 模式 2：内置自检（Built-in Self-Check）

把自检要求放在主指令中，让 AI 在输出时一并完成：

```
"请写一份技术方案。在方案的最后，附带一个【自我审查】部分：

【自我审查】
1. 这个方案最大的 3 个风险是什么？（诚实回答，不要避重就轻）
2. 有哪些部分我用了'通常''一般'这类模糊表达？（列出并修正）
3. 如果团队只有 3 个人，方案中哪些步骤需要简化？
4. 这个方案中有没有我其实不太确定的部分？（标注出来）
5. 如果让我重新写一遍，我会在哪些地方做出不同的选择？"
```

这种内置自检让 AI 在一个回复中完成了"生成+审查"两个步骤，效率更高。

---

#### 模式 3：对抗式反思（Adversarial Reflection）

让 AI 扮演反对者，攻击自己的输出：

```
"刚才你给出了一个产品方案。现在，请你扮演这个方案的最强反对者——一个经验丰富、言辞犀利的 CTO，他最讨厌'过度设计'和'为了技术而技术'。

请从这个视角，逐条批评你刚才的方案：
1. 哪些部分过度设计了？
2. 哪些决策可以用更简单的方式实现？
3. 有哪些隐藏的成本你没算进去？
4. 如果只给一半的预算和时间，你会砍掉什么？

批评完之后，基于这些批评，给出方案的改进版。"
```

---

### 自我反思 vs 反复迭代

你可能会想：这不就是多改几轮吗？不完全是。

```
反复迭代（被动）：
你发现问题 → 告诉 AI → AI 修改
（依赖你自己的判断力，你可能也看不出来问题）

自我反思（主动）：
AI 主动审视 → AI 发现问题 → AI 主动修正
（AI 帮你找问题，你的角色从"挑毛病的人"变成"审核 AI 的发现"）
```

**两者配合使用效果最好**：
1. 主指令 → AI 输出
2. AI 自检 → 发现问题并修正
3. 你审查 → 确认修正是否到位
4. 如果需要 → 再追加一轮你的反馈

## 技巧组合：五合一实战

### 场景：设计一个新产品功能的技术方案

**这是一个重要任务**，所有五个技巧都用上。

```
【指令开始】

🎭 角色扮演：
你是一个在大型互联网公司工作了 12 年的首席架构师。
你经历过 5 次系统大重构，其中有 2 次成功、3 次失败。
你对"技术选型过于激进"有深深的警惕——你见过太多用新技术
最后被迫回退的案例。
你的设计哲学：简单 > 先进，可运维 > 功能多。
你的语言风格：直接、务实，用数字说话，不画饼。

🧠 思维链（分析框架）：
请按以下步骤思考：
Step 1：明确当前系统的限制（是什么让我们不得不改？）
Step 2：列出技术约束（团队能力、现有技术栈、SLA 要求）
Step 3：提出 3 个可选方案（从保守到激进）
Step 4：逐个评估每个方案的优劣（用数据和场景，不要凭感觉）
Step 5：推荐最佳方案，并说明为什么不选另外两个

🔒 约束驱动：
- 方案必须用 Python/Go/Node.js 中的一种（团队只会这三种）
- 必须兼容现有的 REST API 接口（不能改外部协议）
- 新系统 P99 延迟不能超过 100ms
- 部署复杂度：一个新人入职 2 周内能参与开发
- 最多引入 1 个新的基础设施组件（数据库/消息队列等算 1 个）
- 禁止引入 K8s、Serverless、GraphQL（团队没经验，别冒险）

🔄 多轮规划：
这是第 1 轮——先确定技术选型和大方向。
我会根据你的建议，在第 2 轮深入某个方案。
在第 3 轮做细节打磨。

🔍 自我反思（包含在输出中）：
请在方案最后附加：
1. 你用经验（而非理论）否决了哪些常见的方案？为什么？
2. 你对这个推荐方案最大的担心是什么？
3. 有没有"你其实不确定"的部分？标注出来。
4. 如果 3 个月后证明这个方案选错了，最可能的失败原因是什么？

【背景信息】
产品：一个实时数据 Dashboard，当前日均请求量 50 万，
但架构是 3 年前的，现在每次部署都要停服 2 分钟。
团队：6 个后端工程师，都有 Python 背景。
目标：支持零停机部署，日均请求量承载能力提升到 500 万。

【指令结束】
```

### 这个指令的威力

| 技巧 | 在这个指令中的作用 |
|------|-------------------|
| 🎭 角色扮演 | 设定了务实、保守、有经验的架构师视角，避免不切实际的建议 |
| 🧠 思维链 | 规定了 5 步推理流程，确保分析是结构化的 |
| 🔒 约束驱动 | 用 6 条硬约束锁死了方案空间，AI 不会推荐不切实际的技术 |
| 🔄 多轮规划 | 告诉 AI 这是第 1 轮，管理预期——不用一次写完所有细节 |
| 🔍 自我反思 | 要求 AI 标注自己不确定的部分，暴露隐藏风险 |

## 技巧选择指南

| 你的任务特征 | 推荐的主要技巧 | 辅助技巧 |
|-------------|--------------|---------|
| 逻辑复杂，容易出错 | 🧠 思维链 | 🔍 自我反思 |
| 需要特定专业视角 | 🎭 角色扮演 | 🔒 约束驱动 |
| 输出格式/质量要求高 | 🔒 约束驱动 | 🔍 自我反思 |
| 涉及多步、多维度 | 🔄 多轮对话优化 | 🧠 思维链 |
| 结果非常重要 | 五合一 | 五合一 |
| 想要发现自己的盲点 | 🔍 自我反思（对抗式） | 🎭 角色扮演 |
| 需要大量高质量内容 | 🔒 约束驱动 | 🔄 多轮（先对齐标准） |

## 关键要点总结

1. **高级技巧的本质是"控制 AI 怎么想"**——不再是告诉 AI 你要什么，而是控制它的思考过程、视角、边界
2. **思维链**：不要让 AI 直接给答案，要求展示推理过程。最简单的用法只需加一句"让我们一步步思考"
3. **角色扮演**：不只是"你是一个专家"，而是构建完整的人物设定——身份、思维、语言、关注点、价值观
4. **约束驱动**：用精确的边界替代模糊的愿望。约束控制在 5-8 条，多了就排优先级
5. **多轮对话优化**：每轮一个明确目标，一个维度，告诉 AI 保留什么
6. **AI 自我反思**：让 AI 做自己的审查者。生成模式和审查模式是两种不同的状态
7. **五合一**：最重要的任务，大胆用上全部五个技巧——它们不冲突，而是层层叠加

## 实践活动

### 练习 1：给你的下一个指令加上思维链（15 分钟）

选一个你接下来要问 AI 的逻辑分析类问题，加上思维链：

1. 先按你平时的方式写指令并发送
2. 再在同一个问题中加入"请按以下步骤思考：Step 1……Step 2……"
3. 重新发送，对比两次 AI 输出的推理质量差异

### 练习 2：构建一个深层角色（20 分钟）

为一个你常用的场景构建一个深层角色设定：

1. 选一个场景（如代码审查、技术写作、产品分析）
2. 按五维度构建角色：身份背景、思维模式、语言风格、关注焦点、对受众的假设
3. 每个维度至少写 3 条具体设定（不要"有经验"，要"8 年后端，主导过 3 次架构重构"）
4. 用这个角色完成一次实际任务，和不用角色的结果对比

### 练习 3：将"建议"改写为"约束"（10 分钟）

把以下模糊建议改写为精确约束：

1. "请写得有趣一点" → 约束：_______
2. "分析得全面一点" → 约束：_______
3. "代码要写得好" → 约束：_______
4. "风格要专业" → 约束：_______

### 练习 4：让 AI 反思一次（15 分钟）

1. 让 AI 完成一个中等复杂度的任务
2. 在 AI 输出后，用"事后自检"模式让它审查自己的输出
3. 观察：AI 发现了哪些你在第一遍没注意到的问题？

### 练习 5（挑战）：五合一实战（40 分钟）

选一个你手头真实的重要任务，按本课"五合一实战"的格式设计一个完整的指令。要求：
- 角色扮演：构建 5 维度的深层角色
- 思维链：列出 5-7 步推理步骤
- 约束驱动：至少 5 条精确约束
- 多轮规划：说明这是第几轮、后续轮的安排
- 自我反思：内置自检问题

完成后发给 AI，记录首次输出的质量和迭代次数。
