### 2. 指令库的三个层次

指令库不只是"存一堆指令"。它有三个递进的层次：

```
┌──────────────────────────────────────────────────────────┐
│            指令库三层次                                    │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  📦 第 1 层：存储层（"把东西放进去"）                       │
│     有地方存指令，能找到，能复用                            │
│     → 这是基础。没有这一层，一切都是空谈                     │
│                                                          │
│  📊 第 2 层：管理层（"知道什么好用什么不好"）                 │
│     能评估指令质量，能淘汰过时的，能优化表现差的              │
│     → 这是核心。只存不管 = 垃圾场                           │
│                                                          │
│  🚀 第 3 层：增值层（"越用越好，越积越值钱"）                │
│     指令之间有链接、有组合、有进化历史                       │
│     新指令基于旧指令改进而来，而不是每次从零开始              │
│     → 这是终极目标。你的指令库变成你的"数字智慧"             │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**大部分人的"指令库"停留在第 1 层**——建了个文件夹，扔了一些指令进去，然后就不再维护。结果半年后打开，里面的指令已经过时了，还不如重新写。

**你要做的是达到第 3 层**——让你的指令库成为一个活的、成长的、持续增值的体系。

### 3. 五个评估维度：怎么判断一个指令"好不好"？

在把一个指令放入你的指令库之前，需要先评估它。不是所有指令都值得保存——只保存那些经过验证的。

#### 维度 1：有效性（Effectiveness）

**核心问题**：这个指令能稳定地产出符合预期的结果吗？

```
评估方法：
用同一个指令测试 10-20 个不同输入，统计有多少次的输出"可以直接使用
或只需微调"。

评分标准：
🟢 优秀（≥80%）：绝大多数输出直接可用
🟡 合格（60-79%）：需要一定调整但方向总是对的
🔴 差（<60%）：输出不稳定，经常需要大幅修改

如果评分是 🔴，不要存入指令库。先优化，优化后重测，达到 🟡 再入库。
```

**提升有效性的方法**（回顾以前的课程）：
- 加正例和反例（课程 07）
- 明确成功标准（课程 11 错误 7 的改正）

---

#### 维度 2：效率（Efficiency）

**核心问题**：用这个指令比不用 AI，节省了多少时间？

```
评估方法：
计算时间节省比 = (传统方式耗时 - AI 辅助耗时) / 传统方式耗时

评分标准：
🟢 优秀（≥70%）：节省了大部分时间
🟡 合格（40-69%）：有明显节省但还有改进空间
🔴 差（<40%）：投入产出比不够好

注意：不是所有指令都以"省时间"为目标。
有的指令是为了"提高质量"——比如代码审查指令，
它省的不是你的时间，而是帮你发现你可能漏掉的问题。
对于这类指令，用"质量提升度"来评估。
```

**示例**：

```
指令：竞品分析报告生成

传统方式：
  手动收集资料：2 小时
  整理和分析：3 小时
  撰写报告：2 小时
  总计：7 小时

AI 辅助：
  写指令 + 调整参数：10 分钟
  AI 生成初稿：2 分钟
  核实数据 + 补充细节：1.5 小时
  润色和定稿：30 分钟
  总计：约 2.2 小时

效率 = (7 - 2.2) / 7 = 68.6% → 🟡 合格
（还有优化空间：如果给 AI 更好的背景信息，核实时间还能减少）
```

---

#### 维度 3：一致性（Consistency）

**核心问题**：多次使用这个指令，输出的格式、结构、质量是否稳定？

```
评估方法：
同一个指令，用 5 个不同但同类的输入，对比输出：
- 结构是否一致？
- 风格是否一致？
- 质量是否稳定？（没有一次特别好、一次特别差）

评分标准：
🟢 优秀：5 次输出在结构和风格上高度一致
🟡 合格：大体一致，但有些细节波动
🔴 差：每次输出格式都不同，质量忽高忽低

一致性差的常见原因：
- 指令中的约束不够具体（"写一篇好文章" → 每次"好"都不一样）
- 缺少格式模板（AI 每次自由发挥）
- 参数设计不合理（参数太多导致每次组合不同）
```

**提升一致性的方法**：
- 用约束驱动锁定格式（课程 12 技巧 3）
- 提供固定模板而不是自由格式（课程 09）

---

#### 维度 4：易用性（Usability）

**核心问题**：别人（包括 3 个月后的你自己）能轻松使用这个指令吗？

```
评估方法：
让一个不熟悉这个指令的人（或 3 个月没用它的你自己）试着用一次：
- 他能在 5 分钟内看懂这个指令是干什么的吗？
- 他知道参数怎么填吗？
- 他需要问你多少个问题才能成功运行？

评分标准：
🟢 优秀：不熟悉的人 5 分钟内能用，不需要求助
🟡 合格：10 分钟内能用，需要看使用说明
🔴 差：看了说明也不知道怎么用，必须有人教

经验法则：如果你 3 个月后打开这个指令，能直接拿来用 → 🟢。
如果 3 个月后你自己都看不懂了 → 🔴，需要重写说明。
```

---

#### 维度 5：维护成本（Maintenance）

**核心问题**：这个指令需要多频繁地更新？每次更新需要多少工作量？

```
评估方法：
估算过去 3 个月这个指令更新了多少次，每次花了多少时间。

评分标准：
🟢 优秀：每季度更新 1 次以内，每次 < 30 分钟
🟡 合格：每月更新 1-2 次，每次 < 1 小时
🔴 差：每周都要改，或者单次修改需要超过 2 小时

维护成本高的常见原因：
- 指令依赖的外部信息频繁变化（如依赖特定 API 文档）
- 指令设计得太具体，稍一变化就要大改
- 指令写得过于复杂，改一处牵动全身
```

---

#### 五维评估总表

把五个维度的评分汇总，决定指令的命运：

| 总分 | 决策 |
|------|------|
| 🟢🟢🟢🟢🟢 全优 | ⭐ 核心资产，定期维护 |
| 🟢🟢🟢🟢🟡 大部分优 | ✅ 保留，针对性改进 🟡 维度 |
| 混合 🟢🟡 | ✅ 保留，重点改进 🟡 和 🔴 |
| 出现 2+ 个 🔴 | ⚠️ 需要大幅改造或考虑淘汰 |
| 出现 3+ 个 🔴 | 🗑️ 建议淘汰或完全重写 |

**重要的是**：不是每个指令都要追求全优。高频使用的指令（每周多次）值得追求全优；低频使用的（每季度才用一次）达到 🟡🟡🟡🟡🟡 就够了。

### 4. 建立指令库的 7 步法

#### Step 1：审视和收集

不要从零开始。你已经有"种子"了——你过去几个月用过的所有有效指令。

```
审视清单：

□ 打开你的 AI 对话历史（最近 2-3 个月）
□ 标记每条指令的结果质量：✅（好）/ 🟡（还行）/ ❌（差）
□ 筛选出所有 ✅ 和 🟡 的指令
□ 按使用频率排序：找出你用了 3 次以上的指令
□ 这就是你的"种子库"——通常 5-15 条
```

**如果对话历史太多**：只回顾最近 30 天的。重点是找到那些"反复出现"的指令模式——你每次都写了差不多的指令。

---

#### Step 2：精选核心指令

从种子库中选择 5-10 条最高频、最高质量的指令，作为第一批入库。

```
精选原则：

1. 优先高频：每周用 3 次以上的优先
2. 优先高价值：省时最多的、质量提升最明显的优先
3. 优先可复用：参数化程度高的优先（一个模板能覆盖多种场景）
4. 暂时搁置：低频的、一次性的、还在实验中的——先不入库

建议第一批：
□ 3-5 条工作高频指令（你每天都在用的）
□ 1-2 条"救命"指令（偶尔用但用了能省大量时间）
□ 1-2 条"质量"指令（不是为了省时，而是为了保证质量）
```

---

为每条指令建立统一的文档格式。这是指令库"可维护"的基础——格式不统一，找都找不到。


#### Step 4：测试和打分

用五维模型对每条入库指令做一次正式评估：

```
对每条核心指令：
1. 用 3-5 个真实场景测试（不是造出来的测试用例，是你实际工作中遇到的）
2. 记录每次的输出质量和你的调整工作量
3. 按五维模型打分
4. 记录改进方向

记录在指令文档的"效能评估"部分。
```

#### Step 5：组织成库

选择存储方式，建立目录结构：

**存储方式选择**：

| 方式 | 适合 | 优缺点 |
|------|------|--------|
| **Git 仓库** | 个人+团队 | ✅ 版本控制 ✅ 协作 ❌ 需要 Git 基础 |
| **本地文件夹 + 云同步** | 个人 | ✅ 零门槛 ✅ 随时访问 ❌ 无版本控制 |
| **Notion / 飞书** | 团队 | ✅ 易编辑 ✅ 分享 ❌ 无版本控制 |
| **Obsidian / Logseq** | 个人 | ✅ 双向链接 ✅ 知识图谱 ❌ 学习曲线 |

**推荐组合**：本地文件夹（日常快速访问）+ Git 仓库（版本控制 + 备份）

#### Step 6：建立维护节奏

指令库不是建完就完事的。它需要维护：

```
┌──────────────────────────────────────────────────────────┐
│            指令库维护节奏                                  │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  🔄 每次使用后（30 秒）                                    │
│     □ 输出满意吗？不满意的话——记下来哪里需要改              │
│     □ 发现新的使用场景了吗？——补充到"适用场景"              │
│                                                          │
│  📅 每月（30 分钟）                                        │
│     □ 回顾本月使用最多的 3 条指令——还够用吗？                │
│     □ 看看有没有"这个指令我连续 2 个月没用过"——考虑淘汰      │
│     □ 更新效能数据（有效性、效率）                          │
│                                                          │
│  📅 每季度（2 小时）                                       │
│     □ 深度优化 2-3 条高频指令                              │
│     □ 清理：淘汰不再需要的、合并重复的                      │
│     □ 补充：这个季度有没有新的高频任务需要建指令？           │
│     □ 重新评估所有指令的五维评分                           │
│                                                          │
│  📅 每年（半天）                                           │
│     □ 大版本升级：重构过时的指令                           │
│     □ 知识归档：过去一年学到的新技巧，有没有反映在库中？     │
│     □ 结构调整：现在的分类方式还合理吗？                    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**维护的核心原则**：80% 的维护精力放在 20% 的高频指令上。那些你天天用的指令，值得花时间打磨到 🟢🟢🟢🟢🟢。那些半年才用一次的，能用就行。

#### Step 7：分享与进化

一个只存在于你电脑上的指令库，价值有限。分享带来三个好处：

**好处 1：外部反馈**
别人用你的指令可能会遇到你没想到的问题——这些问题就是优化方向。

**好处 2：知识交换**
你分享一个指令，别人可能回馈你一个更好的版本，或者一个你没想到的使用场景。

**好处 3：建立声誉**
在团队或社区中分享高质量的指令，是建立"AI 使用专家"个人品牌的最佳方式。

```
分享的三种方式（渐进式）：

方式 1：一对一分享
  "这个指令我用了很久，效果很不错。你也试试？"
  → 低风险，快速反馈

方式 2：团队内分享
  在团队频道分享 + 案例说明
  → 建立团队影响力

方式 3：公开发布
  GitHub、博客、社区
  → 获得最大范围的反馈和声誉
```

### 5. 指令库常见误区与对策

| 误区 | 为什么有害 | 对策 |
|------|----------|------|
| **贪多求全** | 库里有 100 条指令，但 80 条质量差、不常用、自己也忘了是干嘛的 | 保持核心库在 20-30 条。质量 > 数量。每新增一条，考虑是否淘汰一条 |
| **只存不改** | 指令过时了还留在库里，使用者（包括你自己）被误导 | 每个指令标注"最后更新日期"。超过 6 个月没更新的指令，打开时自动提醒"检查是否过时" |
| **格式混乱** | 有的指令只有一句话，有的写了 10 页。查找成本高 | 强制使用统一的文档模板（本课 Step 3）。格式不一致的不入库 |
| **缺少示例** | 只有指令文本，没有输入输出示例——3 个月后你自己都忘了怎么用 | 每个指令至少配 1 个真实示例。这是强制要求 |
| **孤岛化** | 指令之间没有链接，看不到它们的关系 | 在每个指令文档中加上"相关指令"链接。使用能双向链接的工具（Obsidian/Logseq）效果更好 |
| **维护拖延** | "下个月再整理"→永远不整理 | 把维护放进日历。每月最后一个周五下午 4:00-4:30 = 指令库维护时间 |

### 6. 从指令库到知识体系

当你的指令库积累到一定程度（20+ 条高质量指令），你会发现一个有趣的现象：**指令之间开始产生关联。**

```
示例：
竞品分析指令 ──→ 输出竞品对比表
                     │
                     ├──→ 产品定位指令（基于竞品差异调整定位）
                     │         │
                     │         └──→ 营销文案指令（基于定位写文案）
                     │
                     └──→ 技术方案指令（了解竞品技术栈后选型）
```

**这就是指令库从第 2 层（管理层）升级到第 3 层（增值层）的标志**——指令不再是孤立的，它们形成了工作流和知识网络。

**促进"增值"的实践**：
- 在每个指令中维护"相关指令"链接（双向链接更好）
- 标注指令之间的输入/输出关系（"A 的输出是 B 的输入"）
- 创建"工作流"文档——描述一组指令如何串联完成一个完整的任务

## 关键要点总结

1. **指令库是你最重要的 AI 资产**：它把你学到的知识从"容易遗忘的记忆"变成"永久可用的系统"
2. **三个层次递进**：存储层（有地方放）→ 管理层（知道好坏）→ 增值层（越用越值钱）
3. **五维评估模型**：有效性、效率、一致性、易用性、维护成本。不是每个指令都要全优——高频的值得打磨，低频的能用就行
4. **7 步建立法**：审视收集 → 精选核心 → 标准化文档 → 测试打分 → 组织成库 → 建立维护节奏 → 分享与进化
5. **维护比创建更重要**：不维护的指令库比没有指令库更糟（它会给你过时的、错误的指引）
6. **质量 > 数量**：15 条高质量指令 > 50 条半成品。不求多，求精
7. **持续学习是终身的**：15 课是地基，不是终点。AI 在进化，你也要进化
