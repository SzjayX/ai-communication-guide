# 课程 11：常见错误与防坑指南

## 学习目标
- 识别指令设计中最常见的 10 个错误及其根本原因
- 理解每个错误的"为什么会犯"和"怎么改"
- 掌握错误诊断框架，能自行判断一个指令哪里有问题
- 学会使用防坑检查清单，在发送指令前完成自检
- 能从"凭感觉"的指令写作者，升级为"有方法"的指令诊断者

---

## 核心概念

### 1. 为什么要学"错误"？

前面 10 节课，我们一直在学"怎么做对"。你掌握了：

```
✅ RTBSF 框架（课程 06）      → 搭骨架
✅ 示例驱动（课程 07）         → 给参照
✅ 迭代式沟通（课程 08）       → 持续打磨
✅ 可复用模板（课程 09）       → 固化经验
✅ 系统化思考（课程 10）       → 先想清楚
```

**但你有没有发现一个问题？**

知道怎么做对 ≠ 不会做错。

就像一个学会了所有游泳技巧的人，第一次下水还是可能呛水。因为理论告诉你"手应该怎么划"，但没告诉你"哪些动作会让你沉下去"。

**这节课就是从"呛水"中总结出来的避坑指南。** 每个错误背后，都有真实踩过坑的人。学完这节课，你能在错误发生之前就认出来、绕过去。

---

### 2. 错误诊断框架：先分类，再对症

在逐一讲解 10 个错误之前，先建立一个诊断框架。所有指令错误都可以归入以下四个大类：

```
┌──────────────────────────────────────────────────────────┐
│              指令错误四大类别                              │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  🔴 信息类错误（Information Gap）                         │
│     AI 缺少必要的上下文，被迫猜测                          │
│     └─ 错误 1、4、8、9                                   │
│                                                          │
│  🟡 表达类错误（Clarity Gap）                              │
│     需求说得不够清楚，或内部自相矛盾                        │
│     └─ 错误 2、5、6                                      │
│                                                          │
│  🟠 期望类错误（Expectation Gap）                          │
│     对 AI 的能力、行为、输出有不符合实际的期望              │
│     └─ 错误 3、7、10                                     │
│                                                          │
│  🟢 流程类错误（Process Gap）                              │
│     缺少检验和停止机制，导致反复打转                        │
│     └─ 贯穿多个错误                                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**明白了分类，你就能在遇到问题时快速定位**：是信息不够？表达不清？期望不对？还是流程出问题？

### 3. 10 个常见错误详解

---

#### ❌ 错误 1：信息不足，让 AI 蒙着眼猜

**所属类别**：🔴 信息类错误

**典型场景**：

```
你：帮我写一份推广方案
AI：（写了 800 字，关于电商产品的推广方案）
你：不是，我们的产品是面向开发者的 SaaS 工具
AI：（重新生成，这次对了产品方向，但风格太正式）
你：受众是个人开发者，不是企业，语气轻松一点
AI：（再生成一版，还是不太对）
你：算了，我自己写吧
```

**为什么会犯**：
- 你脑子里有完整的画面——产品是什么、受众是谁、要什么风格
- 你觉得"这么明显的事不用说吧"
- 但实际上 AI 脑子里一片空白，它只能从 9 个字里去猜

**心理学解释**：这叫做**知识的诅咒（Curse of Knowledge）**。一旦你知道了某件事，你就很难想象"不知道"是什么感觉。你对产品的了解是 100%，但你只给了 AI 1% 的信息，却期望它产出 100% 的质量。

**✅ 改正方法**：

给 AI 的信息至少要覆盖以下三个维度：

```
产品或主题的基本信息：
├─ 它是什么？（一句话定位）
├─ 给谁用的？（目标用户/受众）
└─ 核心优势/特点是什么？（3 个以内）

任务的具体要求：
├─ 什么类型的内容？（方案/文案/代码/分析）
├─ 用来干什么？（发布、汇报、说服、学习）
└─ 发表在什么地方？（影响风格和深度）

风格和约束：
├─ 什么语气？（正式、轻松、幽默、专业）
├─ 多长？（字数或篇幅范围）
└─ 有什么绝对不能要/不能碰的？
```

**改正后的指令**：

```
帮我写一份产品推广方案。

产品信息：
- 产品：DevFlow，一个面向开发者的 API 调试工具
- 和 Postman 的区别：内置 AI 自动生成测试用例
- 当前阶段：刚上线 2 个月，有 200 个免费用户
- 定价策略：个人免费，团队版 $9/月

任务要求：
- 类型：线上推广方案（不是广告文案，是渠道+内容策略）
- 目标：3 个月内获得 500 个注册用户
- 预算：3000 元以内
- 团队：我和另一个同事（两个人，都是技术背景，没有市场经验）

约束：
- 我们没时间做视频/播客，只能做文字内容
- 不能花钱投广告，预算主要用于工具订阅和少量推广
- 语气要开发者友好，不要营销腔

请给出：渠道选择 → 内容规划 → 发布节奏 → 效果衡量方式
```

**效果对比**：
- 修改前：3-5 轮迭代，每次都在补充信息
- 修改后：AI 首次就能给出针对性的方案，最多微调

**一句话教训**：**你觉得"不用说"的，往往是最需要说的。**

#### ❌ 错误 2：要求太抽象，没有可衡量的标准

**所属类别**：🟡 表达类错误

**典型场景**：

```
你：帮我写得好一点
AI：（改写了一遍，换了一些词）
你：不是这种"好"，是更有深度的那种
AI：（又改了一版，更长了）
你：也不是越长越好吧……算了
```

**问题**：你用的词——"好"、"深"、"有趣"、"创新"、"专业"——每个人理解都不一样。你心里的"好"可能指"逻辑更严密"，AI 可能理解成"用词更华丽"。

**常见的模糊词及其歧义**：

| 模糊要求 | AI 可能的理解（之一） | 你实际可能想要（之一） |
|----------|---------------------|----------------------|
| "写好一点" | 换更高级的词汇 | 结构更有逻辑 |
| "深入一点" | 写得更长 | 加入因果分析和原理 |
| "有趣一点" | 加笑话和表情 | 用生动的类比和故事 |
| "专业一点" | 堆砌术语 | 引用权威来源和数据 |
| "创新一点" | 说一些没人见过的概念 | 用新的角度分析旧问题 |
| "简洁一点" | 写得短 | 删掉冗余，保留核心 |

**✅ 改正方法**：把形容词翻译成具体的、可执行的要求。

把"好"拆解成具体的维度：

```
❌ "写得好一点"

✅ "请在以下方面改进：
1. 逻辑结构：从'现象 → 原因 → 影响 → 方案'四个层次递进
2. 数据支撑：超过 50% 的观点要有具体数据或案例佐证
3. 可读性：每个段落不超过 4 句话，避免超过 30 字的句子
4. 行动导向：每个建议都要包含'谁、做什么、什么时候完成'"
```

**翻译规则**：

```
模糊词 → 具体标准

"更好"     → "在 ___ 维度上从 ___ 变成 ___"
"更深"     → "加入 ___ 层分析（原因/影响/方案/原理）"
"更专业"   → "使用 ___ 领域的标准术语，引用 ___ 类来源"
"更有趣"   → "用 ___ 个类比/故事/案例来说明"
"更简洁"   → "删到 ___ 字以内，每个观点不超过 ___ 句话"
```

**一句话教训**：**模糊的形容词是 AI 的猜谜游戏。你不说清楚，它就猜，猜不对就该你改。**

#### ❌ 错误 3：过度依赖 AI，放弃了自己的判断

**所属类别**：🟠 期望类错误

**典型场景**：

```
你：这个技术方案可行吗？
AI：完全可行，建议采用微服务架构 + Kubernetes
你：好的，那就按这个做
（3 个月后，你发现你的小项目根本不需要微服务和 K8s）
```

**为什么会犯**：
- AI 的回答很自信，听起来头头是道
- 你对 AI 的领域不够了解，无法判断对错
- 你下意识地认为"AI 说的应该有道理"

**AI 的"盲目自信"问题**：
- AI 的表述永远是自信的——即使它在胡说
- 它不会主动说"我不确定"（除非你要求它）
- 它倾向于给出"全面"的方案，而不是"最合适"的方案

**✅ 改正方法**：把 AI 从"决策者"降级为"参谋"。

**每个 AI 建议都要过这 4 道关**：

```
┌─────────────────────────────────────────┐
│        AI 建议审查四步法                   │
├─────────────────────────────────────────┤
│                                         │
│  Step 1: 相关性检查                      │
│  "这个建议符合我的实际情况吗？"            │
│  ├─ 团队规模和预算匹配吗？                │
│  ├─ 时间线合理吗？                       │
│  └─ 技术/资源限制满足吗？                 │
│                                         │
│  Step 2: 可行性检查                      │
│  "我们能实际执行这个建议吗？"              │
│  ├─ 需要什么前置条件？我们具备吗？          │
│  └─ 有没有更简单但同样有效的方案？          │
│                                         │
│  Step 3: 风险检查                       │
│  "按这个建议做，最坏会发生什么？"           │
│  ├─ 有没有被忽略的风险？                  │
│  └─ 失败的成本我们能承受吗？               │
│                                         │
│  Step 4: 替代方案检查                    │
│  "有没有 AI 没想到的选项？"                │
│  ├─ 完全相反的方案会怎样？                 │
│  └─ 什么都不做会怎样？                    │
│                                         │
└─────────────────────────────────────────┘
```

**你甚至可以直接让 AI 帮你做这些检查**：

```
"对于你刚才给出的方案，请从以下角度重新审视：
1. 相关性：如果团队只有 3 个人、预算只有 5 万，这个方案还合适吗？
2. 风险：这个方案最大的 3 个风险是什么？有没有被低估的风险？
3. 替代方案：如果完全不做你说的事，我们还可以做什么？

请诚实回答——如果你不确定某个部分，直接说出来。"
```

**一句话教训**：**AI 是你的副驾驶，不是你的司机。方向盘永远在你手里。**

#### ❌ 错误 4：假设 AI 记得之前的对话

**所属类别**：🔴 信息类错误

**典型场景**：

```
（上周的对话）
你：我在做一个电商项目，技术栈是 React + Node.js
AI：好的，有什么需要帮忙的？

（今天，开了新对话）
你：帮我写一个购物车组件
AI：（生成了一个用 Vue 写的购物车组件）
你：不是说了我是 React 吗？！
```

**为什么会犯**：
- 人的社交本能——在和人对话时，之前说过的事对方会记住
- 忘了 AI 的"记忆"机制：即使在同一个对话里，上下文窗口也是有限的
- 在不同对话、不同会话之间，AI 完全失忆

**AI 的"记忆"真相**：

```
同一对话内：
├─ 上下文窗口内（通常几十万字）→ ✅ 记得
└─ 超出上下文窗口 → ❌ 不记得（早的部分被"遗忘"）

不同对话/新会话：
└─ 完全不记得 → ❌ 全新开始，什么都不记得
```

**✅ 改正方法**：永远把"关键信息"放在当前指令里。

不要这样做：
```
（依赖上一轮的背景）
"那按刚才说的，帮我继续完善第三部分"
```

而要这样做：
```
"我们之前在讨论 React 电商项目的购物车组件。
需求是：支持增减数量、实时计算总价、库存校验。
请帮我完善第三部分：库存不足时的 UI 提示和降级方案。"
```

**关键信息检查**：每次指令中检查以下内容是否包含：

```
□ 我的身份/角色？（开发者？产品经理？学生？）
□ 当前项目的核心信息？（技术栈、产品类型、阶段）
□ 这次任务的约束？（时间、预算、格式要求）
□ 和之前对话的关系？（"在上次讨论 X 的基础上"）
```

**对你特别重要的场景**：
- 在 Claude Code 中开新对话时 → 重新提供关键背景
- 使用模板时 → 确保参数填全了（模板不知道你是谁）
- 换了一个 AI 工具时 → 从零开始，全部重新说

**一句话教训**：**把每次指令当作你们第一次说话。AI 不会记住你说过什么，直到你在这条消息里告诉它。**

#### ❌ 错误 5：指令内部自相矛盾

**所属类别**：🟡 表达类错误

**典型场景**：

```
你：写一篇深度分析，但不能超过 500 字
（又要有深度，又要很短——深度分析通常需要足够的篇幅展开）

你：语气要幽默有趣，但要保持专业严肃
（幽默和严肃在大多数情况下是矛盾的）

你：请全面覆盖所有方面，但要简洁
（全面 ≠ 简洁，这两者是互斥的）
```

**为什么会犯**：
- 你有多个需求，但没意识到它们之间存在冲突
- 你在一条指令里塞了太多要求，没排优先级
- 你用了两个日常觉得"差不多"但实际冲突的词

**常见的矛盾组合**：

| 要求 A | 要求 B | 为什么矛盾 |
|--------|--------|-----------|
| 详细全面 | 简洁简短 | 全面需要篇幅，简短限制篇幅 |
| 幽默风趣 | 专业严肃 | 幽默降低专业感，严肃排斥幽默 |
| 创新突破 | 参考竞品 | 突破意味着不一样，参考意味着要像 |
| 技术深度 | 通俗易懂 | 深度需要术语和细节，通俗需要回避术语 |
| 与众不同 | 符合行业惯例 | 不同就不能符合惯例 |

**✅ 改正方法**：明确优先级，而不是同时要求。

**方法 1：排优先级**

```
"请写一篇产品分析。要求按优先级排列：
1. [最重要] 分析必须有数据支撑，每个结论要有依据
2. [其次] 篇幅控制在 800 字左右（实在需要展开可以到 1200 字）
3. [再次] 风格偏轻松，可以用类比，但不要用段子或表情"
```

**方法 2：给出具体边界，而不是矛盾标签**

```
❌ "既要有深度又要简洁"
✅ "只分析最核心的 3 个点，每点写透（200 字/点），不追求面面俱到"
```

**方法 3：主动问 AI 处理方式**

```
"以下两个要求可能有冲突：详细分析 vs 1000 字以内。
如果冲突了，优先保证分析的深度和质量，字数可以放宽到 1500。
如果字数真的很重要，请选择最重要的 3 个点深入分析。"
```

**一句话教训**：**如果你同时要 A 和非 A，AI 只能满足其中一个。告诉它优先哪个。**

#### ❌ 错误 6：要求 AI 输出"不存在的格式"

**所属类别**：🟡 表达类错误

**典型场景**：

```
你：用"螺旋递进式叙事结构"写这篇文章
AI：（生成了一段看起来像那么回事的文字，但和你期望的完全不一样）
你：不对，螺旋递进不是这样的……
AI：（又换了一种理解，还是不对）
```

**问题**：你心里有一个结构/格式/风格，但你没有用 AI 能精确理解的方式描述它。你用了自己编的名字，或者一个小众术语，AI 只能根据字面意思去猜。

**哪些情况最容易出问题**：

```
□ 自创术语："用'三明治反馈法'来写"
  → AI 不知道你的"三明治"具体指什么

□ 小众框架："按 XX 公司内部的设计规范来"
  → AI 训练数据里没有你们公司的内部规范

□ 模糊的比喻："写得像《三体》那种感觉"
  → "感觉"太主观，AI 不知道你指的是哪方面的感觉

□ 混合风格："一半是教程一半是散文"
  → 两种风格的配比和切换方式不明确
```

**✅ 改正方法**：

**方法 1：用知名术语（如果适用）**

```
❌ "用三明治反馈法写"
✅ "用'具体表扬 → 建设性建议 → 鼓励收尾'的结构写（也叫反馈三明治法）"
```

**方法 2：详细描述你想要的结构**

```
不要只说一个名字，而要描述结构：

"请按以下结构写这篇技术复盘：
1. 事故时间线（什么时间发生了什么，3-5 条）
2. 根因分析（5 Whys，逐层追问到最深的原因）
3. 直接修复（上线了哪些改动，用时多久）
4. 长期改进（需要做但还没做的，按优先级排列）
5. 经验总结（用 1 句话概括这次最大的教训）"
```

**方法 3：给出一个示例**

```
"我想要的是这种结构（示例）：

【发生了什么】
一句话概括事故，2 月 15 日下午 2 点，支付接口全部返回 500

【为什么发生】
直接原因：数据库连接池耗尽
深层原因：慢查询没有超时控制，拖死了所有连接
根因：没有对查询做性能监控和熔断

【我们做了什么】
14:05 发现 → 14:10 定位 → 14:20 重启数据库 → 14:25 恢复

【以后怎么做】
1. 给所有查询加超时（2 秒）
2. 加入慢查询监控
3. 每季度做一次故障演练

请按同样的结构写我们上周的服务宕机事故。"
```

**一句话教训**：**别让 AI 猜你的"黑话"。要么用大家都知道的术语，要么详细描述。**

#### ❌ 错误 7：只说要什么，没说什么算"对"

**所属类别**：🟠 期望类错误

**典型场景**：

```
你：帮我生成 5 个活动创意
AI：（生成 5 个创意）
你：这不太对啊……不是这种感觉
AI：（又生成 5 个）
你：还是不对……第三个有点接近，但其他都不行
```

**问题**：你给了 AI 一个开放性问题，但没有给它判断标准。AI 只能碰运气——生成 100 个创意，总有 1-2 个碰对，但你得先看完 98 个不行的。

**这不是 AI 的问题，是你的问题**：你心里其实有标准（不然你不会说"不太对"），但你没有提前说出来。

**✅ 改正方法**：提问时就给出"对"和"不对"的标准。

**标准公式**：正例标准 + 反例标准 + 数量/范围

```
"帮我生成 5 个活动创意。

✅ 好的创意应该满足：
- 线上可以执行（不需要线下场地和物料）
- 单人即可组织（不需要跨团队协调）
- 预算 < 500 元（主要是奖品费用）
- 目标受众是 25-35 岁的程序员
- 活动周期在 1 周以内

❌ 不要这样的创意：
- 需要请外部嘉宾/KOL（我们没资源）
- 纯抽奖/转发（和我们产品没关系）
- 黑客松/编程比赛（太重了，不适合我们）
- 任何需要用户付钱的活动

如果这 5 个约束太紧，找不到 5 个，就告诉我最多能找到几个，
不要为了凑数而降低标准。"
```

**效果**：
- 修改前：10 个创意中 8 个不能用 → 你烦躁，AI 也不知道错在哪
- 修改后：5 个创意中 3-4 个直接可用 → 节省你筛选和反复沟通的时间

**一句话教训**：**不要当"你猜我想要什么"这种游戏的主角。把标准亮出来。**

---

#### ❌ 错误 8：塞太多内容，超出 AI 的"注意力"

**所属类别**：🔴 信息类错误

**典型场景**：

```
你：（粘贴了一个 5 万字的项目文档）
你：请帮我全面分析这个项目的问题和改进建议，从技术架构、
    团队协作、产品设计、商业模式、市场定位到竞争对手策略……
AI：（生成了一个看起来不错的分析，但遗漏了第 3 章的技术架构问题）
```

**问题**：AI 有上下文窗口限制（虽然很大，但不是无限）。当你塞入超长内容并要求"全面分析"时，可能出现：

- AI 关注了开头和结尾，忽略了中间部分
- AI 对每个方面都蜻蜓点水，没有深入
- AI 遗漏了重要但不够"显眼"的信息
- 处理时间长，token 消耗大

**不是说不能用长文档**，而是要用正确的策略。

**✅ 改正方法**：

**策略 1：分解任务**

```
❌ "分析整个 5 万字文档"
✅ "先分析第 1-3 章的技术架构部分"
  "然后分析第 4-5 章的团队和流程问题"
  "最后综合前两次分析的结论"
```

**策略 2：带着问题去读**

```
❌ "帮我总结这个长文档"
✅ "这个文档中关于'性能优化'的部分有哪些建议？
    重点关注数据库层面和缓存层面的方案，忽略前端优化。"
```

**策略 3：先给摘要，再深挖**

```
"我先给你这个项目的核心信息摘要：
- 项目：XX 电商平台，React + Node.js + PostgreSQL
- 当前阶段：运营 2 年，日活 5 万，有性能瓶颈
- 我重点关注：数据库查询效率、API 响应时间、缓存策略

文档全文在后面，请基于以上背景，重点分析这 3 个方面的问题。
如果文档中有和这 3 个方面无关但你认为很重要的发现，也请提出来。"
```

**策略 4：分段 + 追问**

```
"下面是项目的技术架构部分（第 1-3 章）。
请先分析这部分，等我确认后再继续下一部分。"
→ AI 分析
→ 你："好的，继续第 4-5 章"
```

**一句话教训**：**AI 的注意力不是无限的。长任务要分解，大问题要拆小。**

#### ❌ 错误 9：隐含需求没说出来

**所属类别**：🔴 信息类错误

**典型场景**：

```
你：帮我写一封邮件，告诉客户项目延期了
AI：（写了一封邮件，直接说"项目要延期了，抱歉"）
你：这样太直接了吧，客户看了肯定生气……
```

**你没说出来但心里默认的"隐含需求"**：
- 先肯定客户的耐心和配合（不要一上来就说坏消息）
- 解释延期的客观原因（不是我们偷懒）
- 给出新的交付时间（光说延期不说新日期等于没说）
- 提供补偿方案或额外价值（让客户觉得延期也有好处）
- 语气要诚恳但自信（不是低声下气地道歉）

这些你心里都明白——如果不明白你不会说"这样太直接了"。但你没写在指令里。AI 不是人类，它不会自动脑补这些社交常识。

**常见的"你应该懂"但 AI 不懂的隐含需求**：

| 你说的话 | 你的隐含需求 | AI 的理解 |
|---------|-------------|----------|
| "写个产品介绍" | 要突出和竞品的区别 | 写一个通用产品介绍 |
| "帮我拒绝这个请求" | 委婉地拒绝，不伤和气 | 直接写"我们拒绝你的请求" |
| "写一份会议纪要" | 按"结论→负责人→DDL"的格式 | 按时间顺序记录讨论过程 |
| "分析一下竞品" | 分析我们的差距和机会 | 客观罗列竞品功能 |
| "优化这段代码" | 保持原有逻辑不变，只优化性能 | 可能改变实现方式 |

**✅ 改正方法**：问自己一句话——

> "如果我说出这个指令，AI 最可能怎么理解？而我实际想要的是不是一样？"

如果不一样，补上你的"隐含假设"：

```
❌ "帮我写一封邮件，告诉客户项目延期了"

✅ "帮我写一封邮件，告诉客户项目延期了。
背景：
- 原定本周五交付，现在需要延期到下周三（延迟 5 天）
- 延期的客观原因：依赖的第三方支付接口升级，等待审核
- 客户关系：这是我们最重要的客户，不能搞僵

邮件的要点（按这个顺序）：
1. 先感谢客户一直以来的配合和耐心
2. 诚恳告知延期和客观原因（不要找借口，实事求是）
3. 明确新的交付时间和这个时间为什么靠谱
4. 提供补偿：免费延长 1 个月的技术支持期
5. 表达我们对质量的承诺（宁可延期也不交付有问题的版本）

语气：诚恳、自信、不让客户觉得我们在推卸责任
不要：过于低声下气地道歉，显得不专业"
```

**一句话教训**：**把你脑子里默认的东西写出来。AI 没有社会经验，它不会替你"脑补"。**

#### ❌ 错误 10：无限迭代，不懂何时停止

**所属类别**：🟠 期望类错误

**典型场景**：

```
第 1 轮：不对，少了点什么
第 2 轮：好一点了，但第三段还是不太对
第 3 轮：第三段好了，但引言又不行了
第 4 轮：引言好了，但总觉得整体差点意思
...
第 7 轮：我又觉得第一版其实挺好的……
```

**为什么会犯**：
- 你没有一个明确的"这就算完成了"的标准
- 完美主义：总觉得还能更好
- 每改一轮，注意力的焦点在变——修了 A 发现 B 不好，修了 B 发现 C 不好
- 你把迭代当作"追求完美"，而不是"解决差距"

**无限迭代的代价**：

| 成本类型 | 具体表现 |
|---------|---------|
| ⏱️ 时间 | 每轮 3-5 分钟，7 轮就是半小时——你本可以用这个时间做别的 |
| 💰 Token | 长对话消耗大量 token，尤其上下文越来越大时 |
| 🧠 判断力 | 来回改让你丧失对"好坏"的敏感度（看太多遍麻痹了） |
| 😤 情绪 | 挫败感和疲劳感会让你做出更差的决策 |

**✅ 改正方法**：迭代前先定义"完成线"。

**方法：设置 Exit Criteria（退出标准）**

```
在开始任何需要迭代的任务时，先写下来：

"以下条件全部满足即为完成：
✅ 包含了需求中列出的 5 个核心卖点
✅ 标题的吸引力评分（我自己评）达到 7/10 以上
✅ 没有事实性错误（所有的数据、日期、名称都已核实）
✅ 至少一个同事看完说'还行'
⏱️ 最多迭代 3 轮——超过 3 轮还没达到，就取最好的一版"
```

**80 分原则**：

```
质量
  │
100├──────────── 完美（不存在）
  │         ╱
 80├───────●── 80 分（够好就行）
  │      ╱
 60├─────╱   ← 前 80% 的质量只需要 20% 的时间
  │   ╱       后 20% 的质量需要 80% 的时间
 40├──╱
  │ ╱
 20├╱
  └──────────────────→ 投入时间
```

**80 分在大多数场景下已经足够好了。** 尤其是：
- 内部使用的文档
- 初稿（还会给别人修改）
- 非关键场景的任务
- 你只是想快速验证一个想法

**一句话教训**：**完美是拖延症的另一个名字。在开始前定义"什么样算够好"，到了就停。**

### 5. 10 个错误速查表

当你遇到问题时，在这里快速定位：

| 你的症状 | 可能犯了哪个错误 | 快速修正 |
|---------|----------------|---------|
| AI 的输出方向对了但很泛 | 错误 1、2 | 补充产品/受众/场景信息，明确要解决的具体问题 |
| AI 的输出和你期待的完全不同 | 错误 9、6 | 写清楚你的隐含需求，用已知术语或详细描述格式 |
| AI 的输出前后矛盾 | 错误 5 | 找出矛盾的要求，明确优先级 |
| AI 的输出遗漏了重要信息 | 错误 8 | 分段提问，或者先给摘要再深入 |
| AI 给的建议听起来很好但执行不了 | 错误 3 | 补充约束条件，明确你的实际资源和限制 |
| 你已经改了 4+ 轮还没满意 | 错误 10、7 | 定义"完成标准"，发指令前就给出对错标准 |
| AI 在新对话里完全不记得之前的事 | 错误 4 | 在每条指令里复述关键背景 |
| AI 的创意很多但都不是你想要的类型 | 错误 7 | 给出正例和反例，明确"对"和"不对"的标准 |

## 关键要点总结

1. **所有错误都可以归入四大类**：信息类（🔴）、表达类（🟡）、期望类（🟠）、流程类（🟢），建立分类思维可以快速定位问题
2. **信息不足是最常见的错误**：知识的诅咒让你觉得"不用说"，但 AI 不知道你脑子里的画面
3. **模糊的形容词是猜谜游戏**：把"好"翻译成具体的维度、标准、数量
4. **AI 是参谋不是决策者**：每个建议都要过"相关性→可行性→风险→替代方案"四道关
5. **不要假设 AI 记得**：把关键背景写进每条指令，当每次都是第一次说话
6. **矛盾的需求要排优先级**：如果既要 X 又要非 X，告诉 AI 优先哪个
7. **提前定义"对"和"完成"**：发指令时就说清楚什么算好、什么时候可以停
8. **防坑检查清单用起来**：每次发指令前花 30 秒自检，这 30 秒是你对自己时间的尊重

## 实践活动

### 练习 1：诊断你的旧指令（20 分钟）

翻出你最近一周的 5 条 AI 对话记录。对每条指令：

1. 对照本课的 10 个错误，看看犯了哪几个
2. 判断属于四大类中的哪一类
3. 用本课的方法重写其中 2 条
4. 把新旧两条指令发给 AI，对比输出

### 练习 2：模糊词翻译练习（10 分钟）

把以下模糊要求翻译成具体的、可执行的要求：

1. "把这个文案写得更好"
2. "分析得深入一点"
3. "让这个设计看起来更高级"
4. "写一篇有趣的博客"
5. "优化一下这个流程"

### 练习 3：矛盾发现练习（10 分钟）

以下指令中包含内部矛盾。找出矛盾，然后给出修正版：

```
指令 A：
"请写一份非常详细全面的技术分析，但要控制在 500 字以内。
要专业严肃，但又要有趣好读。要创新突破，但要参考行业最佳实践。"
```

```
指令 B：
"帮我写一封催款邮件。语气要强硬让客户知道我们是认真的，
但又不能破坏客户关系，要让对方觉得我们还是很友好的。"
```

### 练习 4：建立你的防坑习惯（15 分钟）

1. 把本课的"防坑检查清单"复制到你的笔记工具或打印出来
2. 接下来一周，每次发复杂指令前过一遍清单
3. 一周后统计：你犯了最多的 3 个错误是什么？针对性地复习对应的课程
