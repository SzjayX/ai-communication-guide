# 课程 05：描述背景与受众（别让它猜）

## 学习目标
- 理解背景信息的重要性
- 学会如何有效描述项目背景
- 掌握受众分析和适配的方法

---

## 核心概念

### 1. 为什么背景很重要？

**一个例子**：

```
❌ "帮我改进这个 API"
AI: "有很多改进方向，我列举一些..."
（但不知道最重要的是什么）

✅ "我们的 API 被手机应用调用，每天 100 万次请求。
   最近用户反馈登录特别慢（5 秒），
   这导致了 20% 的用户放弃登录。
   请帮我优化登录 API。"
AI: "明白了，这是性能关键路径，我的优化建议会重点关注..."
```

**背景信息的作用**：
- 帮助 AI 理解优先级和约束
- 避免 AI 给出不适用的建议
- 提高解决方案的针对性
- 节省沟通和返工时间

---

### 2. 背景信息的五个维度

#### 维度 1：历史背景（History）

**包含内容**：
- 项目的来源和演进过程
- 之前尝试过什么
- 为什么之前的方案没有成功

**示例**：
```
"这个项目始于 2023 年，
当时我们用 Django + PostgreSQL 快速原型验证。
验证成功后开始规模化，
但 Django ORM 在大数据量下出现性能问题。
我们尝试过数据库优化和缓存，
但 CPU 使用率仍然超过 80%。
现在我们考虑重新架构，
用 FastAPI + 异步处理来解决这个问题。"
```

**为什么重要**：
- 避免重复之前失败的方案
- 帮助 AI 理解演进过程中的教训
- 提供调整策略的背景

#### 维度 2：业务背景（Business Context）

**包含内容**：
- 这个项目的商业目标
- 涉及的用户或市场
- 竞争环境和差异化点

**示例**：
```
"我们是一个 B2B SaaS 公司，
主要客户是中型制造企业（100-5000 人）。
我们的竞争对手是 SAP 和 Oracle，
但我们的优势是更便宜更易用。
这个功能（库存管理）是我们与竞争对手的重要差异点。"
```

**为什么重要**：
- 帮助 AI 理解功能的优先级
- 对齐商业目标和技术方案
- 指导 AI 在权衡时的决策

#### 维度 3：技术背景（Technical Context）

**包含内容**：
- 现有的技术栈和架构
- 已有的解决方案和库
- 技术约束和限制

**示例**：
```
"我们的后端是 Node.js + Express，
使用 MongoDB 存储数据。
前端是 React + Redux。
我们托管在 AWS 上，使用 Lambda 处理异步任务。
技术约束：
- 无法改变主要框架（需要与现有代码兼容）
- 数据库迁移成本高
- 团队对 Python 不熟悉，只懂 Node.js 和 Python"
```

**为什么重要**：
- 指导 AI 选择兼容的技术方案
- 避免建议不现实的改进（如"改用 Go"）
- 了解约束条件下的最优方案

#### 维度 4：团队背景（Team Context）

**包含内容**：
- 团队的技术水平和经验
- 团队规模和时间约束
- 团队的学习意愿和能力

**示例**：
```
"我的团队有 3 个 Python 开发者，
都有 2-3 年工作经验，
但没人用过 FastAPI 或异步编程。
我们需要 2 周内上线这个功能。
团队有学习意愿，但时间很紧。"
```

**为什么重要**：
- 指导 AI 选择合适的难度等级
- 建议是否需要学习成本
- 评估时间预期的现实性

#### 维度 5：约束条件（Constraints）

**包含内容**：
- 预算限制
- 时间限制
- 性能要求
- 安全和合规要求

**示例**：
```
"约束条件：
- 时间：必须在 2024 年 Q3 完成
- 预算：总预算 50 万，已花 30 万
- 性能：API 响应时间必须 < 100ms
- 安全：需要符合 SOC2 合规
- 用户数：支持 100 万并发用户"
```

**为什么重要**：
- 指导 AI 在约束下找最优方案
- 避免给出不现实的建议
- 帮助权衡各种因素

### 3. 如何描述背景的模板

使用这个模板确保背景信息完整：

```
【项目背景信息】

【历史背景】
- 项目什么时候开始的？
- 主要的演进阶段是什么？
- 之前尝试过什么，为什么失败了？

【业务背景】
- 主要用户是谁？
- 商业目标是什么？
- 与竞争对手的区别是什么？

【技术背景】
- 现在的技术栈是什么？
- 为什么选择这个栈？
- 有什么技术约束吗？

【团队背景】
- 团队规模和经验水平？
- 主要的技能和知识盲点？
- 学习意愿如何？

【约束条件】
- 时间限制：__
- 预算限制：__
- 性能要求：__
- 其他重要约束：__
```

### 4. 理解和描述受众

#### 什么是受众？

受众是将使用或受益于你的产品/功能/文档的人

**常见的受众类型**：

| 受众 | 特点 | 需要什么 |
|------|------|--------|
| 终端用户 | 非技术背景 | 简单易用，不需要懂细节 |
| 技术开发者 | 技术背景 | 准确的技术细节，API 文档 |
| 产品经理 | 商业思维 | 商业影响，用户体验 |
| 投资者 | 关注回报 | 市场潜力，财务预测 |
| 管理层 | 关注成本效益 | ROI，风险评估 |

---

#### 受众分析框架

**步骤 1：定义受众**
```
主要受众：中国中型制造企业的 CFO 和采购经理
次要受众：企业 IT 管理人员
```

**步骤 2：理解受众的特点**
```
- 教育背景：大学本科，财务或管理专业
- 技术水平：了解一些 IT 概念，但不是专家
- 主要关心：成本降低、流程简化、合规
- 工作压力：很忙，很少有时间学习新系统
- 拒绝理由：太复杂、需要太多培训、实施风险高
```

**步骤 3：调整沟通方式**
```
为这个受众：
- 使用他们熟悉的商业语言，避免技术术语
- 强调具体的商业收益（降低成本多少%）
- 提供完整的培训和支持
- 降低实施风险的保证
```

### 5. 为不同受众调整表述

**同一个产品功能，给不同受众的描述**：

**受众：CEO**
```
"新的库存管理系统将把库存成本降低 15%，
通过减少积压和过期物品。
预期投资回报率在 18 个月内达到 200%。"
```

**受众：采购部门**
```
"你们能通过这个系统看到实时库存，
自动获得补货提醒，
减少采购错误。
平均每个采购员的工作效率提升 20%。"
```

**受众：IT 部门**
```
"系统架构：基于微服务，
可以与现有 ERP 系统集成。
API 接口完整，支持定制开发。
部署在云上，自动伸缩。"
```

## 实战案例

### 案例 1：完整的背景描述

**场景**：要求 AI 帮助设计一个支付系统

**不完整的请求**：
```
"设计一个支付系统"
```

**完整的请求**（包含充分背景）：
```
【项目背景】

【历史】
我们是一个 5 年的在线购物平台，现有 200 万用户。
最初使用第三方支付平台（支付宝和微信），
但手续费高达 2%。
现在我们想自建支付系统来降低成本。

【业务目标】
- 降低支付成本到 0.5%
- 提高支付成功率从 95% 到 99.5%
- 支持更多国家和地区（目前只有中国）
- 实现 1 年内节省 500 万元

【技术现状】
- 后端：Java Spring Boot（微服务架构）
- 数据库：MySQL 主从 + Redis 缓存
- 托管：Kubernetes 在阿里云
- 支付流程：目前通过支付宝官方 SDK

【团队情况】
- 支付团队：3 个高级工程师，有支付系统经验
- 时间：3 个月
- 预算：100 万（包括研发和基础设施）

【约束条件】
- 支付安全要求最高（PCI DSS 认证）
- 必须支持中国所有主流银行
- 必须支持云闪付
- 防诈欺系统
- 必须符合国家政策要求

【问题】
请帮我设计这个支付系统的架构，包括：
1. 系统设计方案
2. 安全考虑
3. 与现有系统的集成方案
4. 风险评估
5. 实施路线图
```

---

### 案例 2：为不同受众的表述调整

同一个优化：减少数据库查询

**给 CEO**：
```
"通过数据库缓存优化，
我们的页面加载速度提升了 60%，
用户体验显著改善，
这有助于提升转化率。"
```

**给产品经理**：
```
"用户反馈页面加载慢是第二大投诉，
我们通过引入 Redis 缓存，
减少了数据库查询，
现在 90% 的页面在 1 秒内加载。
这应该能提升用户满意度。"
```

**给技术团队**：
```
"我们在 Redis 引入了分布式缓存层，
对热点数据使用 LRU 淘汰策略，
设置 TTL 为 1 小时。
主要缓存的是用户主页数据和商品列表。
具体的缓存键设计和失效策略在这个 wiki 里..."
```

## 关键要点总结

1. **背景决定理解**：好的背景让 AI 少猜测，多理解
2. **五个维度要完整**：历史、业务、技术、团队、约束
3. **受众很重要**：不同受众需要不同的表述方式
4. **写下来**：把背景明确写出来，发现更多遗漏
5. **不要过多**：背景有用，但也不要冗长到分散注意力
6. **定期更新**：项目背景会变化，及时更新很重要
