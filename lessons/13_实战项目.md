### 2. 什么是 AI Agent？

先澄清一个常见的误解。

**AI Agent ≠ 一条复杂的指令**

```
一条指令：
"帮我写一篇技术博客" → AI 写一篇 → 结束

一个 Agent：
"每周一早上，自动搜索本周最热的技术话题，结合我的博客定位
和用户兴趣，选出 5 个选题，生成大纲，评估难度，排序推荐。
如果某个选题和已发布文章重复，自动过滤。
如果用户上周对某类文章特别感兴趣，调高相关选题的权重。"
→ Agent 每周自动执行这个多步骤的流程
```

**AI Agent 的定义**：

```
┌──────────────────────────────────────────────────────────┐
│              AI Agent = 角色 + 目标 + 工具 + 流程 + 约束     │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  🎭 角色：Agent 的身份和定位（它"是谁"）                     │
│  🎯 目标：要达成的具体结果（"做成什么"）                      │
│  🔧 工具：能调用的能力和数据源（"能做什么"）                   │
│  🔄 流程：任务的执行步骤（"怎么做"）                          │
│  🚧 约束：不能逾越的边界（"不能做什么"）                      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**和普通 AI 对话的区别**：

| 维度 | 普通 AI 对话 | AI Agent |
|------|-------------|----------|
| 交互模式 | 你来问，AI 来答 | Agent 主动执行多步任务 |
| 任务粒度 | 单次请求 | 多步骤、有先后依赖 |
| 信息获取 | 仅靠你的输入 | 能主动调用工具获取信息 |
| 决策能力 | 给建议，你决策 | 在约束范围内自主决策 |
| 错误处理 | 你发现错误再问 | Agent 能自检并调整 |
| 复用性 | 每次重新写指令 | 一次设计，反复使用 |

### 3. Agent 的三种复杂度级别

不是所有 Agent 都需要做到最复杂。根据需求选择合适的级别：

#### 级别 1：简单 Agent（指令型）

**本质**：一条结构化的、参数化的指令，配上固定的执行流程。

```
示例：周报生成 Agent

输入：本周的关键事件（3-5 条）
流程：
  1. 按"进展→问题→计划→支持"结构组织
  2. 用团队周报模板填充
  3. 检查字数（300-500）
  4. 输出周报

适用：流程固定、不需要外部数据、不需要多轮决策
```

**你用课程 06（RTBSF）+ 课程 09（模板）就能做。**

---

#### 级别 2：中等 Agent（工具型）

**本质**：能调用外部工具获取信息，基于获取的信息做决策。

```
示例：技术选题 Agent

输入：博客定位 + 受众画像
工具：
  - 搜索最近 7 天的技术新闻
  - 查询历史文章（去重）
  - 查询用户评论（了解兴趣趋势）
流程：
  1. 搜索热点 → 2. 过滤重复 → 3. 评估热度 →
  4. 生成大纲 → 5. 难度评估 → 6. 排序推荐

适用：需要外部数据、有一定决策逻辑
```

**你需要在课程 06+09 的基础上，加上课程 12 的约束驱动和多轮对话设计。**

---

#### 级别 3：高级 Agent（自主型）

**本质**：能根据反馈和结果调整自身行为，持续优化。

```
示例：智能内容运营 Agent

除了级别 2 的所有能力外，还能：
  - 追踪已发布文章的实际阅读量和用户反馈
  - 根据历史表现调整选题策略
  - 发现某类选题持续低效后，自动减少该类推荐
  - 每周生成一份"内容策略周报"，反思上周决策

适用：需要持续学习、自适应调整
```

**这需要全部 12 课的知识 + 迭代机制 + 反馈闭环。**

---

**本课聚焦于级别 2**——这是实用价值最高、但复杂度可控的级别。掌握级别 2 后，级别 3 是自然的延伸。

### 5. 完整实战：构建"技术文章策划 Agent"

让我们用五步法，从零到一构建一个真实的 Agent。跟完全程，你就知道怎么构建自己的 Agent 了。

#### 项目背景

你运营一个面向 Python 开发者的技术博客，每周需要策划和发布 3-5 篇文章。目前最大的痛点是：
- 选题靠灵感，不稳定——有时候一周想出 5 个好选题，有时候一个都想不出来
- 经常写了才发现"这个话题上个月写过了"（没有系统化去重）
- 不知道用户到底想看什么——凭感觉选题，阅读量忽高忽低

---

### Step 1：定义 Agent

**不要一上来就写指令。先用课程 10（系统化思考）把需求想清楚。**

#### 1.1 回答核心问题

```
Q1 核心目标：
   每周自动生成 5 个高质量的技术文章选题，附带大纲和优先级排序

Q2 为什么需要：
   - 当前选题效率低（每周花 3-4 小时找选题）
   - 文章质量不稳定（选题决定 50% 的文章质量）
   - 用户留存率下降（连续 3 个月内容同质化）

Q3 约束条件：
   - Agent 运行频率：每周一次
   - 信息来源：公开的技术新闻、GitHub Trending、用户评论
   - 输出格式：结构化的选题报告
   - 不做什么：不自动发布文章，不自动写作全文

Q4 受众（Agent 的"用户"）：
   - 主要用户：你自己（博客运营者）
   - 终端读者：初中级 Python 开发者
   - 博客定位：实用导向，代码驱动，不写纯理论

Q5 已有资源：
   - 历史文章数据库（200+ 篇，有标题、标签、阅读量、评论数）
   - 用户评论数据（最近 3 个月）
   - 竞品博客列表（知道他们在写什么）

Q6 成功标准：
   - 主指标：Agent 推荐的选题中，≥60% 最终被采纳并发布
   - 次指标：从选题到发布的时间从平均 3 天降到 1.5 天
   - 学习指标：采纳的文章平均阅读量不低于手动选题时期
```

#### 1.2 定义输入和输出

```
┌──────────────────────────────────────────────────────────┐
│              输入输出定义                                  │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  输入（每次运行时提供）：                                   │
│  ├── 博客定位：Python 开发者，实用导向，初中级              │
│  ├── 时间范围：未来一周                                   │
│  ├── 数量要求：5 个选题                                   │
│  └── 特殊需求（可选）：本周想多写某个主题（如 AI/ML）       │
│                                                          │
│  输出（Agent 生成的报告）：                                │
│  ├── 5 个选题，每个包含：                                 │
│  │   ├── 选题名称（吸引人的标题提案）                      │
│  │   ├── 选题理由（为什么这个值得写）                      │
│  │   ├── 热度评估（基于搜索量/讨论度，高/中/低）           │
│  │   ├── 与历史文章的重复度检查                           │
│  │   ├── 完整大纲（3-5 个小标题 + 每部分核心要点）         │
│  │   ├── 预估写作时间                                    │
│  │   ├── 目标关键词（SEO）                                │
│  │   └── 推荐发布日                                      │
│  ├── 选题间的协同关系（哪些可以做成系列）                  │
│  └── 本周内容策略的总建议                                 │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### Step 2：设计工具和工作流程

#### 2.1 工具清单

Agent 需要的不是真实代码（那是工程师的事），你需要定义每个工具的**用途、输入、输出**：

```
工具 1：搜索技术热点
  用途：获取最近 7-14 天的热门技术话题
  输入：关键词（可选），时间范围
  输出：话题列表 + 热度指标 + 来源链接
  为什么需要：Agent 需要知道最近在流行什么

工具 2：查询历史文章
  用途：查看过去发布过的文章，避免选题重复
  输入：关键词或标签，数量限制
  输出：匹配的文章列表（标题、标签、发布日期、阅读量）
  为什么需要：去重——不能推荐已经写过的选题

工具 3：分析用户评论
  用途：从评论中提取用户感兴趣的话题和未满足的需求
  输入：时间范围，数量限制
  输出：高频话题词 + 用户显式提出的选题建议 + 情感倾向
  为什么需要：用户说出来的需求是最强的选题信号

工具 4：生成文章大纲
  用途：为给定选题生成详细大纲
  输入：选题名称 + 目标受众 + 文章类型
  输出：3-5 个小标题 + 每部分核心要点 + 建议的代码示例
  为什么需要：大纲的可操作性决定了选题是否值得写

工具 5：评估写作难度
  用途：估算写一篇特定文章需要的时间和技术深度
  输入：选题名称 + 大纲
  输出：难度等级（简单/中等/困难）+ 预估时间（小时）
  为什么需要：帮助运营者规划一周的工作量
```

#### 2.3 异常情况处理设计

一个好的 Agent 不是"假设一切顺利"，而是"知道出问题时怎么办"：

```
异常 1：搜索工具返回的热点太少（< 5 个有意义的话题）
  → Agent 行为：扩大搜索时间范围到 30 天，或者在用户评论中多挖掘
  → 输出时标注：本周热点较少，以下选题部分基于用户长期兴趣

异常 2：某选题的大纲生成质量差（太笼统，无法直接使用）
  → Agent 行为：降低该选题的推荐优先级，标注"大纲需人工完善"
  → 如果 ≥ 3 个选题的大纲不合格，重新生成

异常 3：5 个选题中有 2 个和历史文章高度重复
  → Agent 行为：自动过滤，从候选池中递补。如果候选池 < 5 个，
    标注"本周优质选题不足，建议关注以下方向以拓宽选题面"

异常 4：用户评论数据极少（新博客，没有历史评论）
  → Agent 行为：降低用户评论的权重，更多依赖搜索热点和竞品分析
  → 输出时标注：缺少用户反馈数据，选题偏向行业趋势
```

### Step 3：编写 Agent 的完整指令

现在，把前面的设计转化为可执行的指令。**这不是给 AI 的一次性指令，而是 Agent 的"系统提示词"——每次运行都会用的核心指令。**

```markdown
# 技术文章策划 Agent · 系统指令

## 🎭 角色定义

你是一个有 8 年经验的技术内容策略师。你曾为多家知名技术博客
（如 freeCodeCamp、Real Python、CSS-Tricks）策划内容策略。

你的核心信念：
- 好的技术文章不是"某个技术很厉害"，而是"我读完能用它解决什么问题"
- 选题质量决定文章质量的 50%，不要用写作弥补烂选题
- 数据比直觉可靠——热度、用户反馈、历史表现是选题的核心依据
- 对初中级开发者来说，"能不能跑通代码"比"理不理解原理"更重要

你的语言风格：直接、务实、用数字说话。不说"这个选题不错"，
要说"这个选题基于最近 7 天 GitHub 上增加了 1200 个 Star 的 X 项目，
预计能带来 800+ 阅读。"

## 🎯 任务目标

每周生成一份包含 5 个高质量选题的内容策划报告。
每个选题必须附带可直接使用的大纲和评估数据。

## 🔧 可用工具

以下工具由系统提供，你可以调用：

1. `search_trends(keywords, days)` — 搜索最近 N 天的技术热点
2. `get_history(limit, tags)` — 查询历史文章（标题、日期、标签、阅读量）
3. `get_comments(limit, days)` — 获取用户评论数据
4. `generate_outline(topic, audience)` — 生成文章大纲
5. `estimate_difficulty(topic, outline)` — 评估写作难度和时间

## 🔄 工作流程

严格按以下 4 个 Phase 执行：

### Phase 1：信息收集
- 调用 search_trends（关键词：Python、后端开发、AI/ML、
  开发者工具、编程教程，时间范围：14 天）
- 调用 get_history（最近 50 篇，提取标签和主题）
- 调用 get_comments（最近 100 条，提取用户提及的话题和需求）
- 如果某类数据缺失（如没有足够评论），标注原因并调整权重

### Phase 2：选题生成
- 综合分析三个数据源，识别"热点 × 用户兴趣 × 博客定位"的交集
- 生成 10-15 个初始候选选题
- 对每个候选与历史文章做相似度检查，过滤相似度 > 70% 的候选
- 保留 8-10 个进入评估阶段

### Phase 3：评估与精选
- 对每个候选调用 generate_outline 和 estimate_difficulty
- 按以下权重打分（满分 10）：
  · 热度（30%）：话题的搜索量和讨论度
  · 独特性（25%）：和历史文章及竞品文章的差异度
  · 用户需求匹配（25%）：是否直接回应用户评论中的需求
  · 可执行性（20%）：大纲是否具体、写作时间是否在 3-8 小时内
- 排序，选择 Top 5

### Phase 4：输出报告
```

### 6. 好 Agent 的六个特征

基于这个实战案例，我们可以提炼出好 Agent 的通用特征：

| 特征 | 说明 | 你在这个案例中如何实现 |
|------|------|----------------------|
| **目标清晰** | 能一句话说清 Agent 要达成什么 | "每周生成 5 个高质量选题+大纲" |
| **约束明确** | 知道不能做什么，边界在哪 | 6 条约束锁死了选题范围和质量 |
| **工具恰当** | 有完成任务所需的数据和能力 | 5 个工具覆盖了信息获取→加工→输出 |
| **流程合理** | 步骤有逻辑顺序，可追踪 | 4 个 Phase 从粗到细，环环相扣 |
| **优雅降级** | 数据不足或出错时不会崩溃 | 4 种异常场景都有明确的处理策略 |
| **自我审视** | 能检查自己的输出是否合格 | 输出中包含 Agent 自检清单 |

### 7. 常见 Agent 失败模式与对策

| 失败模式 | 表现 | 根因 | 对策 |
|---------|------|------|------|
| **目标漂移** | Agent 的输出越来越偏离初衷 | 角色定义太模糊，缺乏约束 | 强化角色定义，增加必须满足的硬约束 |
| **工具不足** | Agent 反复说"我需要更多信息" | 工具设计阶段遗漏了关键数据源 | 从工作流程反推所需的工具，画流程图检查 |
| **指令冲突** | Agent 行为不稳定，时而 A 时而 B | 指令中存在矛盾要求（课程 11 错误 5） | 逐条检查指令，找出冲突点，排优先级 |
| **输出膨胀** | Agent 的输出越来越长，但信息密度越来越低 | 缺乏格式约束和字数限制 | 加格式模板，明确定量约束 |
| **过度拟合** | Agent 在测试场景表现好，真实场景翻车 | 测试用例太少，没覆盖边界情况 | 设计 3+ 个不同类型的测试用例 |
| **退化遗忘** | Agent 更新后，之前能处理好的场景反而处理不了 | 迭代时只测了新场景，没回归测试 | 每次迭代都用所有历史测试用例复测 |

```
│  ├── 代码审查 Agent                                       │
│  ├── 技术方案评估 Agent                                   │

│  ├── 周报生成 Agent                                       │
│  ├── 会议纪要 Agent                                       │
│  └── 项目风险评估 Agent                                   │
```

**Agent 体系建设原则**：
- 先建最高频使用的（每天/每周都用）
- 每个 Agent 保持独立，但指令风格统一（都用 RTBSF 框架）
- 共享工具和模板，避免重复造轮子
- 定期回顾 Agent 的使用数据，淘汰不用的，优化常用的

## 关键要点总结

1. **Agent 不是复杂指令，是指令系统**：它包含角色定义、工具清单、工作流程、约束条件、输出格式五个层面
2. **五步构建法**：定义 → 设计 → 编写 → 测试 → 迭代。不要跳步——想清楚再写，写好要测，测完要改
3. **从级别 2 开始**：中等复杂度的工具型 Agent 是实用价值最高、学习曲线最平滑的起点
4. **异常处理是区分"能用"和"好用"的关键**：花时间设计 3-5 个异常场景的处理策略
5. **测试要系统化**：3+ 测试用例，覆盖正常、特殊需求、数据稀少三种场景
6. **前 12 课的知识全部用上**：RTBSF（课程 06）、示例（课程 07）、迭代（课程 08）、模板（课程 09）、系统化思考（课程 10）、防坑（课程 11）、高级技巧（课程 12）
7. **Agent 是资产，不是一次性的**：建好后它会持续为你工作。值得花时间打磨
