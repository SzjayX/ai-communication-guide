# 课程 02：大语言模型的能力边界（别神话它）

## 学习目标
- 理解 LLM 的工作原理和根本限制
- 学会识别和规避常见的 AI 错误类型
- 掌握提高准确率的技巧

---

## 核心概念

### 1. LLM 的工作原理：概率补全

**简化的工作流程**：
```
输入文本 → 分词 → 概率计算 → 逐字生成 → 输出文本

示例：
输入："北京是中国的__"
AI: 看这个位置最可能出现什么词？
    → "首都" (概率 85%)
    → "最大城市" (概率 8%)
    → "中心" (概率 5%)
→ 选择概率最高的
→ 输出："首都"
```

**关键理解**：
- AI 不是"理解"，而是"预测下一个最可能的词"
- AI 生成的每个词都是基于前面的文本和统计模式
- 这导致了一系列根本性的限制

---

### 2. 五大常见错误类型

#### 错误 1：幻觉（Hallucination）

**定义**：AI 编造不存在的信息，说得像真的一样

**原因**：当模型遇到不知道的东西时，会基于概率"凑"出答案

**例子**：
```
Q: "请推荐一个 2024 年的新科幻电影"
A: "《量子星河》是 2024 年的科幻大片，由导演李明执导..."
   （这个电影根本不存在！）
```

**如何识别**：
- 让 AI 提供具体来源（如果提供的来源不能验证，可能是幻觉）
- 对重要信息进行交叉验证
- 观察 AI 是否自相矛盾

**如何规避**：
- 明确要求 AI"如果不确定就说不确定"
- 提供足够的背景信息让 AI 参考
- 要求 AI 引用具体来源

**好的指令**：
```
"列出 Python 3.11 的新特性。如果你不确定某项特性是否存在，
请明确标注（[不确定]）。请不要编造特性。"
```

---

#### 错误 2：上下文混淆（Context Confusion）

**定义**：AI 混淆了不同话题的信息或遗忘了之前说过的话

**原因**：LLM 的上下文窗口有限，长对话中易丢失早期信息

**例子**：
```
Q1: "我的项目使用 React"
Q2: "还有其他框架吗？"
Q3: "...对话很长后..."
Q4: "我的项目最适合哪个框架？"
A: "Vue 很好，Svelte 也不错..."
   （忘了你一开始就在用 React！）
```

**如何规避**：
- 重要信息重复提及
- 关键假设明确陈述
- 定期提醒 AI 你的核心背景

**好的指令**：
```
"我的项目基于 React（这很重要，请记住）。
现在我遇到了一个性能问题...
请在建议中考虑我使用 React 这个前提。"
```

---

#### 错误 3：过度自信（Overconfidence）

**定义**：AI 对错误的答案也表现出很高的确定性

**原因**：AI 的生成方式使得它对所有答案都表现出相似的信心水平

**例子**：
```
Q: "2+2等于多少？"
A: "2+2等于4，这是明确的数学事实。"  ✅ 正确且自信

Q: "全球最聪明的动物是什么？"
A: "全球最聪明的动物是海豚，这是科学共识。"
   ❌ 虽然有一定依据，但表现得太绝对了
   （其实这是开放性问题，有多种合理答案）
```

**如何规避**：
- 对于开放性问题，要求 AI 列出多个观点
- 明确要求 AI 表达不确定性（用"可能"、"通常"等词）
- 对高风险决策要求 AI 标注置信度

**好的指令**：
```
"解释这个算法的时间复杂度。
请用以下方式标注你的置信度：
- [高] = 确定无疑
- [中] = 相当确定但可能有例外
- [低] = 这是我的最佳猜测，需要验证"
```

---

#### 错误 4：遗忘或简化

**定义**：AI 遗忘了细节或过度简化了复杂概念

**原因**：
- 上下文长度限制，需要选择保留什么信息
- 某些信息在训练数据中不常见

**例子**：
```
长文档总结后，关键的异常情况被遗漏了
复杂算法讲解时，边界条件被忽略了
```

**如何规避**：
- 提前提醒 AI 重要细节
- 要求 AI 保留特定信息
- 多轮对话中定期检查理解

**好的指令**：
```
"总结这个项目的架构，特别要包括以下关键点：
1. 数据库连接池的配置
2. 错误处理机制
3. 安全认证流程

请确保这三点在总结中清晰呈现。"
```

---

#### 错误 5：格式和结构错误

**定义**：AI 生成的代码、JSON 或其他结构化数据格式不正确

**原因**：虽然 AI 理解结构，但概率生成导致细节出错

**例子**：
```python
# AI 生成的代码缺少引号或括号
data = {name: John, age: 30}  # JSON 格式错误

# AI 生成的 SQL 有语法问题
SELECT * FORM users WHERE id = 1;  # FORM 应该是 FROM
```

**如何规避**：
- 总是验证生成的代码（运行测试）
- 要求具体的格式说明
- 提供格式示例

**好的指令**：
```
"生成一个 Python 函数，返回格式必须是这样的 JSON：
{
  \"status\": \"success\"|\"error\",
  \"data\": [...],
  \"error_message\": \"...\"
}
确保所有引号都正确。"
```

### 3. 模型能力的关键因素

#### 因素 1：模型大小
```
参数量越多 ≈ 越复杂的任务能力

Haiku 3B       → 快速，简单任务
Sonnet 70B     → 平衡，大多数任务
Opus 200B+     → 最强，复杂推理
```

#### 因素 2：训练数据质量
```
高质量数据 → 更准确的输出
广泛的数据 → 更多领域覆盖
最新的数据 → 更少过时信息
```

#### 因素 3：指令微调
```
经过指令微调的模型 → 更好地理解复杂指令
遵循指令更准确 → 更少出现"我做不了"的情况
```

### 4. 模型选择建议

| 任务类型 | 推荐模型 | 原因 |
|---------|--------|------|
| 快速简单任务（翻译、格式转换） | Haiku | 快速且便宜 |
| 编程、分析、创意 | Sonnet | 能力均衡 |
| 复杂推理、深度分析 | Opus | 最强能力 |
| 成本敏感型 | Haiku | 最便宜 |
| 对质量要求极高 | Opus | 最准确 |

## 关键纠正认知

### ❌ 常见误解

1. **"更大的模型总是更好"**
   - ✅ 实际情况：更大的模型能力强，但也更慢更贵。选择合适的即可。

2. **"AI 说不出来的就是不知道"**
   - ✅ 实际情况：AI 可能知道但表达方式不对。换个问法试试。

3. **"AI 的答案就是标准答案"**
   - ✅ 实际情况：AI 的答案只是基于训练数据的最可能答案，需要验证。

4. **"AI 越聪明越不会出错"**
   - ✅ 实际情况：即使最强的模型也会出错，只是错误类型不同。

---

## 关键要点总结

1. **AI 是概率机器**，不是知识库
2. **幻觉是根本问题**，无法完全消除，只能规避
3. **上下文重要**，好的背景信息能显著提升质量
4. **验证必须**，永远不要直接相信 AI 输出
5. **选对模型**，根据任务选择合适的模型
6. **清晰指令**，减少歧义能明显提升准确率
