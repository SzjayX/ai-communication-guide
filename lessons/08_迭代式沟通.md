# 课程 08：迭代式沟通（从初稿到精品）

## 学习目标
- 理解为什么迭代是 AI 沟通的核心技能
- 掌握"初稿→反馈→改进"的系统迭代流程
- 学会给 AI 精准反馈而非笼统抱怨
- 能运用三种迭代模式应对不同场景

---

## 核心概念

### 1. 一个重要的心态转变

**你的第一版指令，几乎是注定不完美的。**

这不是因为你写得不好，而是因为：

- 你脑子里想的 vs 你写出来的，天然有差距
- AI 不是你肚子里的蛔虫，它只能根据文字猜
- 有些细节你只有在看到结果后才会意识到"哦，我其实想要的是另一种"

**高手和普通人的区别不在于第一版指令的质量，而在于迭代的速度和精度。**

```
普通人的模式：
写指令 → AI 输出不满意 → "AI 好笨" → 放弃或接受次品

高手的模式：
写指令 → AI 输出不满意 → 分析问题 → 调整指令 → 再次输出 → 持续优化 → 精品
```

**类比**：做菜
- 新手：严格按菜谱 → 味道不对 → 不知道怎么办
- 大厨：按感觉做 → 尝一口 → 太淡了加盐 → 再尝 → 火候不够再烧 → → 完美

AI 沟通也是如此——**迭代不是失败，是必要的工作流**。

---

### 2. 迭代的五步循环

```
┌──────────────────────────────────────┐
│                                      │
│  ① 写初稿 ──▶ ② 看结果 ──▶ ③ 找问题  │
│                                      │
│      ┌───────────────────────┘        │
│      │                                │
│  ⑤ 微调 ◀── ④ 调整指令               │
│      │                                │
│      └──▶ 满意？──▶ ✅ 完成           │
│              │                        │
│              └──▶ 不满意？──▶ 回到 ②   │
└──────────────────────────────────────┘
```

下面我们用一个完整案例，逐步走完这个循环。

---

#### 第一步：写出初稿指令

假设你要 AI 帮你写一封产品发布邮件。

**初稿指令**：
```
帮我写一封产品发布邮件，告诉用户我们的 App 有新版了。
```

这是典型的"随口一说"级别的指令。让我们看看 AI 会给出什么。

**AI 输出（初稿）**：
```
亲爱的用户：

我们很高兴地通知您，我们的 App 发布了新版本。

新版本包含多项改进和 bug 修复，为您提供更好的使用体验。

请立即更新您的 App，享受最新功能！

谢谢您的支持。
```

---

#### 第二步：审视结果，识别问题

先不急着抱怨"AI 写得烂"，而是**系统地排查问题**。

**对照检查清单**：

| 检查维度 | 当前状态 | 问题？ |
|---------|---------|--------|
| 有没有抓住受众注意力？ | "我们很高兴地通知您"——太套路 | ❌ 没人会读完 |
| 有没有说清楚新功能？ | "多项改进和 bug 修复"——什么都没说 | ❌ 用户不知道更新了什么 |
| 有没有行动号召？ | "请立即更新"——太生硬 | ❌ 没有动力去更新 |
| 有没有品牌个性？ | 完全没有 | ❌ 换谁家的 App 都能用 |
| 语言是否生动？ | 全是套话 | ❌ 像模板生成的 |

**六个检查维度全部不合格。** 但这不是 AI 的错——我们给的指令只有一句话，AI 只能脑补。

---

#### 第三步：调整指令（第一轮迭代）

现在我们知道了问题在哪，开始改进指令。

**改进后的指令**：
```
你是我们 App 的产品经理，负责向用户传达新版本的亮点。

【产品信息】
- App 名称：FocusFlow
- 定位：帮助用户专注工作的番茄钟 App
- 用户画像：25-35 岁职场人士，每天使用
- 新版本亮点：
  1. 新增"深度专注模式"——屏蔽所有通知
  2. 专注数据可视化——看到你一周的专注趋势
  3. 支持与日历联动——自动识别会议时间

【要求】
- 邮件要让人想读完第一段
- 具体说明每个新功能的好处（不是列功能，是讲好处）
- 有一个自然的行动号召
- 语气轻松但专业，像朋友推荐好东西
- 150-200 字
```

**AI 输出（第一轮迭代）**：
```
Hi，你有没有这种感觉——

正要进入专注状态，手机突然弹了 3 条通知，然后
你就忘了刚才在干什么。

FocusFlow 新版本就是帮你解决这个的。三个更新：

🧘 深度专注模式
一键开启，手机变"砖"——通知、来电、App 推送全部静默。
我们测试用户说这是他们"一天中最高效的 2 小时"。

📊 专注趋势图
终于知道自己什么时候最专注了。一周的数据可视化，
帮你找到你的"黄金时间段"。

📅 日历联动
专注时段自动避开你的会议时间。不用手动设置，
FocusFlow 自动帮你规划。

更新到了最新版就能用这些功能。如果你觉得有用，
也欢迎告诉你的同事——好工具值得分享 🙂
```

---

#### 第四步：再次审视

| 检查维度 | 当前状态 | 判断 |
|---------|---------|------|
| 抓住注意力 | "你有没有这种感觉"开头 | ✅ 有共鸣 |
| 说清楚好处 | 每个功能都讲了用户价值 | ✅ 清楚 |
| 行动号召 | "更新到最新版"+"告诉同事" | ✅ 自然 |
| 品牌个性 | 轻松、像朋友聊天 | ✅ 有个性 |
| 语言生动 | 具体场景+用户原话引用 | ✅ 生动 |

**进步显著！** 但还可以调：最后一段的行动号召可以更有力一点，数据可视化那段缺少一个具体例子。

---

#### 第五步：微调到满意（第二轮迭代）

```
很好！请做两个小调整：

1. 专注趋势图那段，给一个具体的例子——
   比如"你会发现每天早上 9-11 点是你最高效的时段"

2. 结尾加一句更有力的行动号召，
   比如让用户更新后回复你这封邮件分享他们的专注时间
```

**AI 输出（最终版）**：
```
Hi，你有没有这种感觉——

正要进入专注状态，手机突然弹了 3 条通知，然后
你就忘了刚才在干什么。

FocusFlow 新版本就是帮你解决这个的：

🧘 深度专注模式
一键开启，手机变"砖"——通知、来电、App 推送全部静默。
测试用户说这是他们"一天中最高效的 2 小时"。

📊 专注趋势图
终于知道自己什么时候最专注了。
比如你会发现"原来每天早上 9-11 点是我最高效的时段"——
然后你就能把最重要的任务安排在这段时间。

📅 日历联动
专注时段自动避开你的会议时间，不用手动设置。

更新到最新版就能用。试试深度专注模式，
然后回复这封邮件告诉我你专注了多长时间——
我很想知道你的最佳纪录是多少 🙂
```

**从初稿到最终版，我们只花了 2 轮迭代。** 但质量的差距是天壤之别。

### 3. 给 AI 反馈的三个层次

很多人不会给反馈，导致迭代效率很低。反馈有三个层次：

#### 层次一：笼统抱怨（无效）

```
❌ "还是不太对"
❌ "感觉不好"
❌ "再写一遍，要好一点的"
```

**为什么无效**：AI 不知道你具体不满意什么。它只能重新随机生成一次，大概率还是不对。

---

#### 层次二：精准反馈（有效）

```
✅ "第二个标题很好，但第一和第三个太像新闻标题了，
   能不能改得更像问题？"

✅ "语气太正式了，我们的受众是创业团队，
   用轻松的口吻，少用'贵司''敬请'这类词"

✅ "例子太少。目前给了 2 个例子，我想要 5 个，
   覆盖不同规模的公司（初创、中型、大型企业）"
```

**为什么有效**：
- 指出了**具体哪里**有问题（不是笼统的"不对"）
- 说明了**为什么**是问题（让 AI 理解你的思维）
- 给出了**改进方向**（让 AI 知道往哪走）

---

#### 层次三：精准反馈 + 示例（最高效）

```
✅ "目前输出的结构是：
   背景 → 方案 → 实施步骤。
   
   我想要的顺序是：
   结论先行 → 方案对比 → 推荐方案 → 实施步骤。
   
   参考这个结构：
   ---
   ## 结论：建议采用方案 B
   ## 方案对比
   [表格]
   ## 为什么选方案 B
   [3 个关键理由]
   ## 实施步骤
   [具体步骤]
   ---"
```

**为什么最高效**：不仅说清楚了问题，还给了 AI 一个"标准答案"作为参考。结合了课程 07 的示例驱动技巧。

### 4. 迭代沟通的三个黄金技巧

#### 技巧一：逐维度迭代，不要一次全改

❌ **一次改太多**：
```
标题要更短、语气要更活泼、
加入数据、改成问题形式、
提到目标受众、避免行业术语……
```
→ AI 会顾此失彼，你不知道哪个改动导致了哪个效果。

✅ **逐维度改进**：
```
第 1 轮：专注长度和清晰度
第 2 轮：调整语气（更专业 or 更有趣）
第 3 轮：加入数据或案例支撑
第 4 轮：检查是否有不必要的术语
第 5 轮：微调措辞和节奏
```

**原则**：每一轮只改一个维度。这样你能清楚地看到每次改动的效果，也更容易定位问题。

---

#### 技巧二：说"为什么"，不只说"改什么"

❌ **只说改什么**：
```
把"最佳"改成"推荐"
把语气改得更正式
```

AI 会机械地执行，但不知道你的意图，容易在其他地方犯同样的错。

✅ **说清楚背后的原因**：
```
把"最佳"改成"推荐"——
因为"最佳"暗示我们做了全面评测，
但实际上我们只对比了 5 个工具。
用"推荐"更准确，不会误导读者。
```

```
语气改得更正式——
因为这份文档会发给公司高管，
他们期望看到严谨的表述，
太口语化会显得不专业。
```

**好处**：当 AI 理解了你的**决策逻辑**，它会在整个文档中一致地应用这个逻辑，而不是只改你指出的那一处。

---

#### 技巧三：好的部分要明确肯定，只调不好的部分

❌ **全部推翻重来**：
```
这个不好，完全重新写
```
→ 浪费了上一轮积累的好内容，下一版可能更差。

✅ **保留好的，调整差的**：
```
前两个段落的节奏很好，保留了。
第三段的数据分析很好，也保留。
但开头的引言太长了，缩短到 2 句。
结尾的行动号召太软了，改得更直接。
```

**一个实用的反馈模板**：
```
✅ 保留：（哪些部分做得好，为什么）
🔧 调整：（哪些部分需要改，怎么改，为什么）
➕ 补充：（哪些是缺失的，需要加什么）
```

### 5. 三种最常见的迭代场景

#### 场景一：数量调整

AI 的输出方向对了，但数量不够或太多。

```
当前输出：3 个 SEO 标题
反馈：方向都对了，但我需要更多选择。
      请给 10 个，分成两组：
      - 5 个偏商业视角（吸引决策者）
      - 5 个偏技术视角（吸引开发者）

当前输出：10 个方案，太长了
反馈：保留最好的 3 个，每个加一段推荐理由
```

---

#### 场景二：风格/语气调整

内容方向对了，但风格不符合受众。

```
当前输出：技术文档的语气
反馈：内容很好，但这份是给 CTO 看的方案建议，
      不需要太技术化。
      
      调整方向：
      - 减少实现细节（如具体代码、API 名称）
      - 增加业务影响分析（成本、时间、风险）
      - 每段开头先说结论，再说细节
```

---

#### 场景三：结构调整

内容可以，但组织方式需要改变。

```
当前输出：按时间顺序讲述
反馈：好的，但我想改成以用户痛点为主线：
      
      新结构：
      1. 用户痛点（我们要解决什么问题）
      2. 解决方案（我们怎么解决）
      3. 证据（数据或案例证明它有效）
      4. 下一步（用户要做什么）
      
      把现有内容按这个结构重新组织。
```

### 6. 迭代中的常见错误

| 错误 | 表现 | 纠正 |
|------|------|------|
| **一不满意就重写** | 每次都从零开始写指令 | 保留好的部分，只调整有问题的 |
| **一次改太多维度** | 同时调整结构、语气、长度、受众 | 每一轮只改一个维度 |
| **不给原因只说结果** | "把 X 改成 Y"但不解释为什么 | 每次都加一句"因为……" |
| **过早停止迭代** | "差不多就行"，但实际差很多 | 用检查清单逐项核实 |
| **过度迭代** | 为了一个词反复调 5 轮 | 80% 满意就够了，完美主义是陷阱 |
| **忘了保存版本** | 覆盖指令后找不回之前的版本 | 保留每次迭代的指令副本 |

### 7. 何时停止迭代？—— 80 分原则

```
┌──────────────────────────────────────┐
│                                      │
│  质量 100 ┤                      ╭── │
│          │                  ╭────╯   │
│       80 ┤          ╭──────╯         │
│          │    ╭─────╯                │
│       60 ┤ ╭──╯                      │
│          │╱                          │
│       40 ┤                           │
│          │                           │
│       20 ┤                           │
│          │                           │
│        0 ┼──┬──┬──┬──┬──┬──┬──┬──   │
│          1  2  3  4  5  6  7  8      │
│                 迭代轮数               │
└──────────────────────────────────────┘
```

**80 分是甜点**：
- 第 1-2 轮迭代提升巨大（从 40 分到 80 分）
- 第 3-5 轮提升放缓（从 80 分到 90 分）
- 第 6+ 轮几乎不提升（从 90 分到 92 分，但花的时间一样多）

**判断是否可以停止**：

| 问题 | 如果答案是"是" |
|------|-------------|
| 输出能直接用吗？ | 不需要你再大改 |
| 比你自己写快吗？ | 节省了时间 |
| 传递了核心信息吗？ | 读者能获取要点 |
| 差的那 20% 影响目标吗？ | 大概率不影响 |

**记住**：追求 100 分的成本可能是 80 分的 5 倍，但收益只有 20%。省下的时间去迭代下一个任务更有价值。

### 8. 迭代与前面课程的关系

#### 结合 RTBSF（课程 06）

每次迭代都可以针对 RTBSF 的五要素逐个调整：

```
第 1 轮：检查 Role（角色定义是否合适？）
第 2 轮：检查 Task（任务描述是否精确？）
第 3 轮：检查 Background（背景信息是否充分？）
第 4 轮：检查 Steps（步骤指导是否清晰？）
第 5 轮：检查 Format（格式要求是否明确？）
```

#### 结合示例驱动（课程 07）

迭代过程中，示例是最有力的工具：

```
第 1 轮：给正例（"我想要这样的"）
第 2 轮：给反例（"不要这样的"）
第 3 轮：给对比（"A 比 B 好在哪"）
```

#### 三课串联，形成肌肉记忆

```
RTBSF 框架（课程 06）
    ↓ 搭建指令骨架
示例驱动（课程 07）
    ↓ 提供具体参考
迭代式沟通（课程 08）
    ↓ 持续调优
高质量输出 ✅
```

## 关键要点总结

1. **心态转变**：不是"一次写对"，而是"逐步调优"。第一版不完美不是失败，是正常。
2. **五步循环**：写初稿 → 看结果 → 找问题 → 调整指令 → 微调。每次只改一个维度。
3. **精准反馈**：不说"不行"，说"第 X 段的问题是什么，因为什么，改成什么方向"。
4. **三层反馈质量**：笼统抱怨 < 精准反馈 < 精准反馈+示例。层次越高，迭代越快。
5. **保留好的部分**：好的部分明确肯定，只调有问题的。不要推翻全部。
6. **80 分原则**：80% 满意的输出就够用了。过度迭代的边际收益极低。
7. **保存迭代过程**：每次迭代都在积累——这些版本就是你的指令库素材。

## 实践活动

### 练习 1：初稿诊断（15 分钟）
写一个简单的指令给 AI，然后分析输出：
- [ ] 用课程 03 的五大标准评估输出
- [ ] 列出 3 个具体的问题
- [ ] 对每个问题，写出改进方向

### 练习 2：三轮迭代实战（30 分钟）
选一个真实工作场景的指令，完成 3 轮迭代：
- 第 1 轮：从模糊指令到清晰指令
- 第 2 轮：调整风格和语气
- 第 3 轮：微调到满意

保存每一轮的指令和输出，对比三轮的进步。

### 练习 3：精准反馈训练（20 分钟）
找一个不满意的 AI 输出，练习写出三个层次的反馈：
1. 笼统抱怨版（你现在会怎么写）→ 意识到这样无效
2. 精准反馈版（指出具体问题 + 原因）
3. 精准反馈 + 示例版（最好的方式）

对比三个版本，体会差距。

### 练习 4：建立迭代检查清单（15 分钟）
根据你在练习 1-3 中的体会，建立你自己的迭代检查清单：
- 我会先检查______
- 然后检查______
- 最后检查______
- 我停止迭代的标准是______
