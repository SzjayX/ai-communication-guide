# 课程 10：AI 思维导图与系统化思考

## 学习目标
- 理解为什么"想清楚再问"是高级指令设计的基础
- 掌握思路检查清单，在提问前系统化地梳理需求
- 学会用思维导图工具分解复杂问题
- 掌握 5 个系统化思考工具并能在实战中应用
- 能将系统化思考的成果转化为高质量的 RTBSF 指令

---

## 核心概念

### 1. 一个被忽视的真相：问题出在提问之前

先来看一个场景。你可能经历过：

```
你：帮我写一份产品推广方案
AI：（写了500字泛泛而谈的方案）
你：不对，我们这个产品是面向开发者的，而且预算很少
AI：（重新生成，但还是差点意思）
你：能不能重点讲一下社区运营的部分？
AI：（第三次生成，终于有点像了）
```

这个过程花了你 15 分钟，迭代了 3 轮。问题出在哪？

**不是 AI 理解能力差，是你的问题本身就没想清楚。**

当你跟 AI 说"帮我写一份产品推广方案"时，你脑子里的画面可能是：

```
产品推广方案
├── 产品：一个 API 调试工具
├── 受众：后端开发者
├── 预算：5000 元
├── 渠道：技术博客 + GitHub
├── 时间：2 周内上线
└── 目标：获得 500 个注册用户
```

但 AI 脑子里没有任何画面。它只能根据你给的 9 个字去猜。猜错了，你来改；再猜，再改。

**这就是本课要解决的核心问题**：在问题出口之前，先把问题想明白。

---

### 2. 核心公式：清晰的思考 → 高质量的指令 → 优秀的 AI 输出

```
你的思维清晰度          AI 输出质量
     ★☆☆☆☆  ──────────→  随机、不可用
     ★★☆☆☆  ──────────→  方向对但细节差
     ★★★☆☆  ──────────→  能用但需要改
     ★★★★☆  ──────────→  首次即基本满意
     ★★★★★  ──────────→  超出预期、可复用
```

这不是玄学，而是一个简单的信息论原理：**AI 是信息放大器，不是信息创造器。** 你输入的信号越清晰，它放大后的输出就越精准。你输入的信号如果模糊，它只能放大模糊。

**系统化思考就是帮你把模糊的念头，变成清晰的信号。**

### 3. 思路检查清单：提问前必答的 6 个问题

在把问题丢给 AI 之前，花 3-5 分钟过一遍这个清单。这 3-5 分钟能帮你省下 3-5 轮迭代。

```
┌─────────────────────────────────────────────────────────┐
│              提问前 · 思路检查清单                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Q1: 核心问题是什么？                                       │
│      └─ 能用一句话说清楚吗？还是混了好几个问题？                  │
│                                                         │
│  Q2: 为什么要解决它？（背景与动机）                             │
│      ├─ 不解决会怎样？                                      │
│      └─ 解决了有什么收益？                                   │
│                                                         │
│  Q3: 有什么约束条件？                                       │
│      ├─ 时间、预算、人力？                                   │
│      ├─ 技术、平台、工具限制？                                │
│      └─ 有什么绝对不能碰的红线？                               │
│                                                         │
│  Q4: 受众是谁？                                            │
│      ├─ 谁会看到/用到这个结果？                               │
│      ├─ 他们的知识水平如何？                                  │
│      └─ 他们最关心什么？                                     │
│                                                         │
│  Q5: 已经有什么想法或方案了吗？                                │
│      ├─ 有哪些可选方案？各自的优缺点？                          │
│      ├─ 有没有倾向的方案？为什么？                             │
│      └─ 有没有参考案例或对标？                                │
│                                                         │
│  Q6: 成功的标准是什么？                                      │
│      ├─ 什么算"好"？什么算"够用"？                             │
│      ├─ 能不能量化衡量？                                     │
│      └─ 有没有可以对照的例子？                                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**重要提示**：不是每个问题都需要回答全部 6 个。简单任务（如"帮我翻译这段话"）可能只需要 Q1+Q4。复杂任务（如"帮我设计一个用户增长策略"）则建议全部过一遍。

### 4. 实战案例：一个问题的三种问法

让我们用一个真实场景来体验"想清楚"和"没想清楚"的差距。

**场景**：你是一家 SaaS 创业公司的技术合伙人，想做内容营销但不擅长。

---

#### 级别 0：没想清楚（模糊提问）

```
帮我写一篇技术博客
```

**这个提问的问题**：
- 什么产品？（AI 完全不知道）
- 面向谁？（写太深新手看不懂，写太浅专家觉得水）
- 什么风格？（教程？观点？案例？）
- 多长？（300 字短文还是 3000 字深度长文？）
- 目的是什么？（引流？品牌？SEO？转化？）

**AI 的困境**：它必须在完全黑暗中摸索。最终给你一篇"万能模板"式的文章——看起来像那么回事，但跟你的实际需求毫无关系。

**结果**：不可用。需要 3-5 轮迭代，甚至可能放弃。

---

#### 级别 1：想清楚了一些（中等提问）

```
写一篇面向后端开发者的技术博客，介绍我们的 API 调试工具。
风格是教程类型，教大家如何用这个工具提高效率。
```

**好了一些**，AI 至少知道：
- 受众：后端开发者
- 主题：API 调试工具
- 风格：教程

**但还是缺了不少**：
- 工具的核心优势是什么？（AI 只能从通用角度夸）
- 有什么具体的功能亮点？（AI 可能写了一堆你不那么重要的功能）
- 竞品是谁？有什么差异化？（AI 不知道你的定位）
- 文章要多长？发在哪？（影响深度和格式）
- 想达到什么目的？（引流注册 vs 品牌曝光，写法完全不同）

**结果**：方向对了但内容不痛不痒。需要 1-2 轮迭代。

---

#### 级别 2：系统化思考后（高质量提问）

先花 5 分钟用思路检查清单梳理：

```
Q1 核心问题：写一篇能带来注册转化的技术博客，推广 APIHub（我们的 API 调试工具）
Q2 为什么：产品刚上线 2 个月，需要高质量技术用户，目前主要靠口口相传，太慢
Q3 约束：预算 0（自己写 + AI 辅助），1 周内发布，目标平台是掘金和 SegmentFault
Q4 受众：后端开发者（中高级），他们对 Postman 很熟悉，对我们不熟悉
Q5 已有思路：我们的核心优势是"AI 驱动的自动测试"，比 Postman 的测试功能强大 10 倍
   参考对标：Postman 的博客风格
Q6 成功标准：文章阅读量 3000+，从文章点击注册的用户超过 20 个
```

然后把梳理结果转化为指令：

```
【角色】
你是一位资深的后端技术博主，擅长写深入浅出的技术教程。
你的读者是工作 3-8 年的后端开发者，对 API 开发非常熟悉，
日常使用 Postman 作为 API 调试工具。

【任务】
写一篇技术教程形式的博客文章，标题为：
"API 自动测试：从手动写断言到 AI 全自动，我每周省了 12 小时"

【背景】
- 产品：APIHub，一个 AI 驱动的 API 调试工具
- 核心差异：AI 自动生成测试用例和断言，覆盖边缘情况
- 竞品情况：Postman 的测试功能需要手动编写，耗时长且容易遗漏
- 目标平台：掘金、SegmentFault（中国开发者社区）
- 当前阶段：产品刚上线 2 个月，需要高质量早期用户

【步骤】
1. 用一个真实痛点开场：手动写 API 测试有多痛苦（共情）
2. 展示传统方式（Postman 手动测试）的流程和时间成本
3. 介绍 AI 自动化测试的原理（不要太技术，讲清楚思路即可）
4. 实操演示：用 APIHub 对一个真实接口做自动化测试
5. 对比总结：手动 vs 自动的时间、覆盖率、可靠性
6. CTA：邀请读者免费试用，附上注册链接

【格式】
- 总长度：2000-2500 字
- 格式：Markdown
- 标题：用数字和具体收益吸引点击
- 必须包含：至少 2 个代码示例、1 个对比表格、1 个思考问题引发评论
- 文末 CTA：简短、自然、不推销，像一个朋友推荐

【示例参考】
我喜欢的博客风格（附上一个你喜欢的技术博客链接或描述风格）：
——读起来像朋友分享经验，而不是官方文档
——用"你"而不是"开发者们"
——先说为什么（Why），再说怎么做（How）
```

**这个指令的威力**：
- AI 知道你的产品、优势、竞品 → 内容精准匹配你的定位
- AI 知道受众和平台 → 风格和深度恰到好处
- AI 知道结构和参考 → 输出结构不会跑偏
- AI 知道成功标准 → 它会朝"带来注册转化"的方向写

**结果**：首次输出即可用，可能只需微调标题或 CTA 文案。

### 5. 三个级别对比

| 维度 | 级别 0（没想清楚） | 级别 1（想了一些） | 级别 2（系统化思考） |
|------|-------------------|-------------------|---------------------|
| 准备时间 | 10 秒 | 1 分钟 | 5 分钟 |
| AI 迭代次数 | 3-5 轮 | 1-2 轮 | 0-1 轮 |
| 总耗时 | 15-25 分钟 | 8-15 分钟 | 6-8 分钟 |
| 输出相关性 | 低（通用模板） | 中（方向对、不痛不痒） | 高（精准匹配需求） |
| 输出可用性 | 需大改或重写 | 需中度修改 | 可直接用或微调 |
| 可复用性 | 不可复用 | 勉强可复用 | 可直接做成模板 |

**反直觉的发现**：系统化思考明明多花了 5 分钟准备，总耗时反而是最少的。因为你把花在"反复改"上的时间，转移到了"一次想清楚"上。**想得越清楚，改得越少，总时间反而更短。**

### 6. 系统化思考的 5 个工具

以下 5 个工具适用于不同场景。不需要每次全都用——根据问题的复杂度和重要性选择。

---

#### 工具 1：5 Why 问题分解（找到根因）

**适用场景**：你在问"怎么办"，但还没搞清楚"为什么"

**为什么重要**：很多人急着问 AI "怎么解决 X 问题"，但其实 X 只是一个表象。如果你没找到根因，AI 给的方案就是治标不治本。

**操作步骤**：对一个问题连续问 5 次"为什么"

**案例**：

```
表象问题：用户注册后 7 天留存率只有 30%

Why 1: 为什么留存率低？
→ 因为用户注册后没有再次回来使用产品

Why 2: 为什么不回来？
→ 因为他们在首次使用时没有体验到产品的核心价值

Why 3: 为什么没体验到核心价值？
→ 因为新用户的引导流程太复杂，核心功能被埋在了第三层菜单

Why 4: 为什么引导流程这么复杂？
→ 因为我们把所有功能都堆在了新用户引导里，想让用户"全面了解"

Why 5: 为什么堆了所有功能？
→ 因为我们没有定义"新用户首日最关键的那个动作"是什么

→ 根本原因：缺乏"关键激活动作"的定义，导致引导流程过于复杂
→ 给 AI 的问题从：
  "怎么提高留存率？"（太宽泛）
  变成：
  "如何设计一个简化的新用户引导流程，只聚焦 1 个关键激活动作，
   让用户在首次使用 3 分钟内体验到产品核心价值？"
```

**5 Why 的注意事项**：
- 不一定是恰好 5 次，关键是问到不能再往下拆为止
- 答案必须有依据，不能凭想象——如果不确定就去查数据
- 如果一个问题有多个原因，分别对每个原因追问

---

#### 工具 2：影响力矩阵（分清轻重缓急）

**适用场景**：你有很多问题想问 AI，不知道从哪个开始

**核心思想**：不是所有问题都值得认真问 AI。有些该问，有些自己搞定就行。

```
                  重要性
                   高
                    │
      ⭐ 计划做       │    ⭐ 立即做
      (重要+不紧急)   │    (重要+紧急)
                    │
      花时间打磨      │    马上去做
      适合深度对话     │    适合精雕细琢
                    │
  ──────────────────┼────────────────── 紧急性
                    │
      📌 排期        │    ⚠️ 快速处理
      (不重要+不紧急) │    (紧急+不重要)
                    │
      可以不做        │    快速搞定
      或模板一把梭    │    别花太多时间
                    │
                   低
```

**使用方式**：

```
本周要做的事：

🔴 立即做（重要+紧急） → 给 AI 最详细的指令，确保一次高质量输出
   □ 给投资人准备下周三的 BP（影响融资）
   □ 修复线上支付 Bug（影响营收）

🟡 计划做（重要+不紧急） → 分步跟 AI 深度对话，可以多轮迭代
   □ 设计下个季度的技术架构升级方案
   □ 建立团队的编码规范文档

🟠 快速搞定（紧急+不重要） → 用模板一把梭，不必追求完美
   □ 回复 5 封普通的商务邮件
   □ 整理本周的会议记录

🟢 排期/不做（不重要+不紧急） → 能不做就不做，或者一键生成
   □ 给内部活动写个简单的通知
   □ 帮你想想团建吃什么（直接问就行）
```

**给你省下的时间**：那些"紧急+不重要"的事，用模板 30 秒搞定，把精力留给真正重要的指令。

---

#### 工具 3：利益相关者分析（全局视角）

**适用场景**：你的决策或方案会影响多个人/多个团队

**为什么重要**：当你说"帮我设计一个方案"时，你心里可能只考虑了自己的需求。但现实中，一个方案可能需要平衡多个利益方的诉求。如果你没想到，AI 也不会主动提醒你。

**操作步骤**：画一个简单的利益相关者地图

```
┌─────────────────────────────────────────────────┐
│              利益相关者分析                         │
├─────────────────────────────────────────────────┤
│                                                 │
│  这个决定/方案会影响谁？                            │
│                                                 │
│  👨‍💻 开发团队                                     │
│     ├─ 影响：增加 2 周工作量                        │
│     ├─ 诉求：技术方案要清晰，别频繁改需求             │
│     └─ 他们最关心的：可维护性、技术债                │
│                                                 │
│  👥 用户                                         │
│     ├─ 影响：获得更好的体验                         │
│     ├─ 诉求：快、稳定、不要改太多                    │
│     └─ 他们最关心的：功能是否解决了他们的问题          │
│                                                 │
│  📊 产品经理                                      │
│     ├─ 影响：决定下个季度的路线图                    │
│     ├─ 诉求：有数据支撑，能衡量效果                  │
│     └─ 最关心的：对核心指标的影响                    │
│                                                 │
│  🏢 管理层                                       │
│     ├─ 影响：需要批准预算和人力                     │
│     ├─ 诉求：清晰的 ROI 和时间线                    │
│     └─ 最关心的：投入产出比、风险                    │
│                                                 │
│  🔧 运维团队                                      │
│     ├─ 影响：维护成本可能增加 20%                   │
│     ├─ 诉求：部署简单、监控完善                     │
│     └─ 最关心的：稳定性和故障恢复                    │
│                                                 │
└─────────────────────────────────────────────────┘
```

**转化为指令**：

不要只跟 AI 说"帮我设计 X 方案"。而要说：

```
我们要做 X 功能，但需要考虑多方诉求：
- 开发团队关心可维护性（不能太复杂）
- 用户想要快和稳定（不能影响性能）
- 产品需要看到数据效果（需要埋点和衡量指标）
- 运维担心维护成本（部署要简单）

请帮我设计一个能在这些约束下找到平衡点的方案。如果有冲突（比如用户要快 vs 开发要可维护），请明确指出并给出你的建议。
```

**这样 AI 的输出会考虑得更全面**，而不是只从一个视角出发。

---

#### 工具 4：约束条件清单（减少 AI 的自由度）

**适用场景**：你有很多限制条件，AI 的自由发挥空间太大

**核心思想**：AI 是一个"没有约束就天马行空"的模型。你需要告诉它边界在哪。边界越清晰，输出越精准。

**约束条件清单**：

```
📅 时间约束
   ├─ 总截止日期：3 个月
   ├─ 第一阶段：2 周内必须有 MVP
   └─ 每周我可投入的时间：15 小时

💰 资源约束
   ├─ 总预算：50,000 元
   ├─ 人力：3 人（2 后端 + 1 前端，没有设计师）
   └─ 外部依赖：需要采购 XX 服务（每月 2000 元上限）

🛠️ 技术约束
   ├─ 必须使用：Python + React + PostgreSQL
   ├─ 不能使用：Kubernetes（运维团队不熟悉）
   ├─ 必须兼容：已有的用户认证系统
   └─ 性能要求：API 响应时间 < 200ms（P99）

📊 质量约束
   ├─ 核心功能可用性 > 99.9%
   ├─ 测试覆盖率 > 80%（核心模块 > 90%）
   └─ 不能降低现有功能的质量

🚫 红线（绝对不能碰）
   ├─ 不能暴露用户隐私数据
   ├─ 不能对现有付费用户造成服务中断
   └─ 不能违反 GDPR 合规要求
```

**把约束转化为指令**：

```
“在以下约束下设计技术方案：
1. 3 人团队（2 后 1 前），没有设计师
2. 2 周内必须出一个可用的 MVP
3. 必须兼容现有的用户认证系统
4. API 响应时间 P99 < 200ms
5. 预算 50k（已经包括了服务器和外采服务的费用）

请先告诉我：在这些约束下，最务实的方案是什么？
如果有些约束互相冲突（比如 2 周 MVP vs P99 < 200ms），请说出来，我们讨论优先级。”
```

**效果**：AI 不会再给你一个"100 人团队+无限预算"的理想方案。它会务实地思考，给出的建议是你能实际执行的。

#### 工具 5：决策树（帮 AI 帮你做选择）

**适用场景**：你在几个选项之间纠结，想让 AI 帮你分析

**核心思想**：把"我该选 A 还是 B"这种主观纠结，变成 AI 能帮你分析的结构化问题。

**操作步骤**：

```
                       该不该重构老系统？
                       /               \
                    重构                不重构
                   /    \              /      \
              风险高    风险低      维持现状   渐进优化
              /    \    /    \      /    \   /    \
           成本高  成本低 ......   慢但稳  累积债 成本低  见效快
```

**把决策树转化为指令**：

不要只说"我该重构吗？"。而要说：

```
我们在纠结要不要重构老系统。请帮我分析：

选项 A：全面重构
├─ 优点：解决长久的技术债，团队士气提升
├─ 缺点：预计需要 3-4 个月，期间新功能开发暂停
├─ 风险：可能引入新 Bug，影响现有用户
└─ 成本：约 2 人年的工作量

选项 B：渐进优化
├─ 优点：不影响业务推进，一边修一边走
├─ 缺点：技术债被稀释但不消失，可能越攒越多
├─ 风险：优化速度赶不上新债产生速度
└─ 成本：持续投入，每月约 30% 人力

请从以下角度帮我评估：
1. 从工程角度看，哪种更可持续？
2. 有没有第三个选择？（比如：先把核心模块重构，其他部分渐进优化）
3. 如果我选了渐进优化，有什么具体的方法可以防止"优化速度赶不上新债速度"？
```

**关键技巧**：不仅列出选项，还要列出每个选项的优缺点和你的判断——AI 才能帮你"算"而不是"猜"。

### 7. 用思维导图整理复杂问题

当问题有多个层次、多个分支时，线性文字很难看清全貌。思维导图是梳理复杂问题的最佳工具。

**推荐工具**：
- **Mermaid**（文本绘图，可以和 AI 协作生成）
- **XMind / MindNode**（传统思维导图软件）
- **Excalidraw**（手绘风格，适合快速草图）
- **直接在对话中画**（用文本缩进，AI 能读懂）

---

#### 案例：用思维导图规划一个内部知识库产品

```
🎯 内部知识库产品设计
│
├─ 用户研究
│  ├─ 工程师（主用户）
│  │  ├─ 痛点：文档散落各处（Notion/GitHub/飞书/口口相传）
│  │  ├─ 习惯：搜索优先，不浏览
│  │  └─ 期望：写一次、到处能搜到
│  ├─ 产品经理（次要用户）
│  │  ├─ 痛点：找不到历史决策和背景
│  │  └─ 期望：按项目聚合、有决策记录
│  └─ 新人（重要受益者）
│     ├─ 痛点：入职头两周完全不知道去哪找信息
│     └─ 期望：按"我要做什么"而不是"文档在哪里"来组织
│
├─ 核心功能（按优先级）
│  ├─ P0：搜索
│  │  ├─ 跨平台搜索（GitHub Issues + Notion + 飞书文档）
│  │  ├─ 语义搜索（不只是关键词匹配）
│  │  └─ 搜索即创建（搜不到时直接创建新文档）
│  ├─ P1：文档协作
│  │  ├─ Markdown 编辑器（技术团队习惯）
│  │  ├─ AI 辅助写作（自动补全、格式检查）
│  │  └─ 版本历史 + Blame（谁写的、为什么这么写）
│  └─ P2：知识图谱
│     ├─ 自动关联相关文档
│     ├─ 专家识别（谁在哪个领域写得最多）
│     └─ 知识更新提醒（这篇文档 3 个月没人维护了）
│
├─ 技术选型
│  ├─ 搜索引擎：Elasticsearch vs Meilisearch vs Typesense
│  ├─ AI 嵌入：用公司已有的 OpenAI 额度
│  └─ 存储：PostgreSQL（全文搜索） + S3（文件附件）
│
├─ 实施路线
│  ├─ Phase 1（3 周）：MVP — 搜索 + 基础 Markdown 编辑器
│  ├─ Phase 2（3 周）：AI 辅助 + 版本历史
│  └─ Phase 3（4 周）：知识图谱 + 专家识别
│
└─ 成功指标
   ├─ 采用率：Phase 1 后 30% 工程师周活
   ├─ 搜索效率：找文档平均时间从 8 分钟降到 2 分钟
   └─ NPS：内部 NPS > 30
```

**用这个思维导图给 AI 提问**：

有了这个思维导图，你给 AI 的问题就从虚无缥缈变成了高度聚焦：

```
❌ 之前（无思维导图）：
"帮我设计一个知识库产品"

✅ 之后（有思维导图）：
"现在我们的知识库产品规划到了 Phase 1（搜索 + Markdown 编辑器），
目标用户是工程师，核心痛点是文档散落在 GitHub/Notion/飞书。
请帮我做以下事情：
1. 设计 Phase 1 的 PRD

4. 设计一个让工程师愿意从'不写文档'变为'写文档'的激励方案"
```

**每个问题都具体、可执行、有上下文。AI 的输出质量必然大幅提升。**

### 8. 系统化思考 + RTBSF = 终极指令框架

现在我们把本课的"系统化思考"和前几课的"RTBSF 框架"结合，形成一个完整的从思考到指令的流程。

```
┌──────────────────────────────────────────────────────────┐
│            从思考到指令：完整流程                            │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  【阶段 1：系统化思考】（本课的核心）                         │
│  ┌──────────────────────────────────────────────┐        │
│  │ Q1 核心问题 → Q2 动机背景 → Q3 约束条件        │        │
│  │ → Q4 受众分析 → Q5 方案思路 → Q6 成功标准      │        │
│  │                                              │        │
│  │ 可选工具：5 Why / 影响力矩阵 / 利益相关者 /      │        │
│  │         约束清单 / 决策树 / 思维导图             │        │
│  └──────────────────────────────────────────────┘        │
│                        ↓                                 │
│  【阶段 2：RTBSF 结构化】（课程 06 的核心）                   │
│  ┌──────────────────────────────────────────────┐        │
│  │ R 角色  ← Q4 受众 + 我需要 AI 扮演什么专家      │        │
│  │ T 任务  ← Q1 核心问题（清晰化后的版本）         │        │
│  │ B 背景  ← Q2 动机 + Q3 约束 + Q5 已有方案      │        │
│  │ S 步骤  ← 基于思维导图/决策树拆解出来的子任务    │        │
│  │ F 格式  ← Q6 成功标准的具体呈现形式             │        │
│  └──────────────────────────────────────────────┘        │
│                        ↓                                 │
│  【阶段 3：增强】（课程 07-09 的核心）                       │
│  ┌──────────────────────────────────────────────┐        │
│  │ + 正例/反例（课程 07）→ 校准质量标准             │        │
│  │ + 迭代意识（课程 08）→ 准备首次结果后的调整方向   │        │
│  │ + 模板化（课程 09）  → 如果这是高频任务，直接套   │        │
│  └──────────────────────────────────────────────┘        │
│                        ↓                                 │
│                 🎯 高质量指令                              │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### 融合示例

需求："我想用 AI 帮我做竞品分析"

```
【阶段 1：系统化思考】（5 分钟）
├─ Q1 核心问题：分析 3 个竞品，找到我们产品可以差异化的切入点
├─ Q2 动机：产品定位不清晰，销售经常被问"你们和 XX 有什么区别"
├─ Q3 约束：3 天内完成，信息来源限公开资料（不爬数据），
│          不需要设计（纯文字分析即可）
├─ Q4 受众：我和另外 2 个联合创始人（全是技术背景）
├─ Q5 已有思路：隐约觉得"AI 原生"是我们的差异，但不确定够不够
│   竞品 A/B/C 都已上市 3+ 年，功能和客户群都稳定
└─ Q6 成功标准：能直接用于下一次融资 BP 的"竞争分析"部分

【阶段 2：RTBSF】
R 角色：资深产品战略分析师，有 10 年 SaaS 行业经验
T 任务：分析 3 个竞品（A/B/C），找出我们有实质性优势的差异化方向
B 背景：← 来自上面 Q1-Q6
S 步骤：
  1. 每个竞品分析：定位、核心功能、目标客户、定价、优劣势
  2. 横向对比表（用对比维度：AI 能力、集成性、易用性、价格、生态）
  3. 找出 3 个我们比竞品强的点（必须是我们能做到而竞品短期难追赶的）
  4. 每个差异化点的可行性评估（技术难度、市场接受度、护城河深度）
F 格式：结构化报告，含对比表格，总长 3000 字左右

【阶段 3：增强】
+ 示例（课程 07）：附上一份你觉得不错的竞品分析报告作为参考
+ 迭代准备（课程 08）：如果有不确定的信息，让 AI 标注出来而非瞎编
+ 模板化（课程 09）：如果经常做竞品分析，做完这版就固化为模板
```

**最终指令**就不在这里完整展示了——它会是 300-500 字的结构化指令，AI 看了就能精准执行。这就是系统化思考的威力：用 5 分钟的准备，换一次到位的结果。

## 关键要点总结

1. **问题出在提问之前**：AI 输出质量的上限，是你思路清晰程度的下限。没想清楚的问题，给 AI 再多上下文也没用
2. **思路检查清单 6 问**：核心问题 → 动机背景 → 约束条件 → 受众分析 → 已有方案 → 成功标准。复杂任务建议全部过一遍
3. **多花 5 分钟想，少花 15 分钟改**：系统化思考看似"多了一步"，实际总耗时更短——因为你把反复修改的时间转移到了想清楚上
4. **5 个工具选着用**：5 Why（找根因）、影响力矩阵（分优先级）、利益相关者分析（全视角）、约束清单（设边界）、决策树（帮选择）
5. **思维导图是复杂问题的解药**：当问题的分支超过 3 层时，先用思维导图梳理，再把梳理结果转化为聚焦的指令
6. **系统化思考 + RTBSF = 终极框架**：思考阶段解决"问什么"，RTBSF 解决"怎么问"，两者结合的指令几乎没有失败的可能
7. **这是一个可以训练的习惯**：刚开始可能觉得"好麻烦"，但用 10 次之后就会变成肌肉记忆。就像健身——前几次酸痛，之后越来越轻松

## 实践活动

### 练习 1：用 6 问清单改造一个问题（15 分钟）

选一个你最近问了 AI 但结果不太满意的问题。用本课的思路检查清单重新梳理：

1. 把原始问题写下来
2. 逐项回答 6 个问题（不知道的写"不知道"——那本身就是问题）
3. 根据梳理结果重写指令
4. 对比新旧两个指令，想想 AI 会给出怎样不同的回答

### 练习 2：给一个复杂问题画思维导图（20 分钟）

选一个你手头正在纠结的复杂问题（工作或生活都可以），画一个思维导图。

要求：
- 至少有 3 层分支
- 包含：背景、方案、约束、利益相关者、成功标准 中的至少 3 个维度
- 工具不限：纸笔、XMind、Mermaid、对话中缩进——都行

画完后，基于思维导图写出 3 个具体的、可以立刻丢给 AI 的问题。

### 练习 3：选一个工具实战（25 分钟）

从本课的 5 个工具中选择一个，应用到你的实际工作中：

- **5 Why**：找一个反复出现的问题，追问 5 层，找到根因
- **影响力矩阵**：把你本周要做的 5-10 件事放进四象限，决定哪些要认真写指令
- **利益相关者分析**：找一个影响多人的决策，画出利益相关者地图
- **约束清单**：为你当前的项目列一个完整的约束清单
- **决策树**：对一个两难选择画决策树，列出优缺点后问 AI

完成后记录：用了哪个工具？有什么新的发现？

### 练习 4：完整走一遍"思考→RTBSF"流程（30 分钟）

选一个你接下来真正需要 AI 帮忙的复杂任务，按本课的完整流程走一遍：

1. 系统化思考（5 分钟）：回答 6 个核心问题
2. 根据需要使用 1-2 个工具（5 Why / 约束清单等）
3. 将思考结果填入 RTBSF 框架（10 分钟）
4. 写一个完整的指令（5 分钟）
5. （可选）加一个正例或反例（5 分钟）

把最终指令发给 AI，观察输出质量。和之前的经验对比——是不是首次输出的质量明显提升了？

## 下堂课预告

📚 **下一课：常见错误与防坑指南**

学会了好方法很重要。但**知道哪些坑不能踩，同样重要。**

下一课，我们把镜头转向"教訓"——那些最常见的指令错误：

- 一口气问太多（AI 不知道该回答哪个）
- 指令自相矛盾（既说要详细又说要简洁）
- 用模糊的形容词（"写得好一点"、"分析得深一点"）
- 把 AI 当搜索引擎（问事实性问题但想要分析性回答）
- 不给反例（只说"我要 X"，没说"我不要 Y"）

每个错误都会附带：真实案例 → 为什么会犯 → 怎么修正 → 修正后的效果。

**准备好了吗？避坑开始！** 🚧
